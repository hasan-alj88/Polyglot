<task type="checkpoint:decision" gate="blocking">
  <decision>Resolve 7 open design gaps for schema properties and dict:K:V</decision>
  <context>These gaps block spec file updates. Each needs a clear ruling before Plan 02 can proceed.</context>
  <options>

    <option id="gap-1">
      <name>Gap 1: Generic param binding syntax</name>
      <pros>
      `{#} #Array<type` defines a generic param. Usage is `;array.int`.
      Question: How does `.int` bind to `<type`?

      **Recommendation: Positional binding.**
      - `;array.int` → 1st param after base = `<type` = int
      - `;array.int:2D` → 1st param = int, 2nd param = 2
      - `;dict:string:int` → 1st param = string (`<keyType`), 2nd param = int (`<valueType`)
      - The separator (`.` vs `:`) comes from the collection's field kind, not from the generic system
      - Rule: params bind left-to-right in declaration order
      </pros>
      <cons>No named parameter binding — positional only. Sufficient for collections (max 2 params) but could be limiting for future complex generics.</cons>
    </option>

    <option id="gap-2">
      <name>Gap 2: `;*` wildcard type</name>
      <pros>
      Used in `[#] <type << ;*` and `[:] :*;*` in collection definitions.

      **Recommendation: `;*` is the "any type" wildcard.**
      - In type parameter defaults: `<type << ;*` means "accepts any type"
      - In field declarations: `:*;*` means "any key, any value type"
      - `:*;type` means "any key, values of `type`"
      - Grammar: `type_ref ::= ... | "*"` (wildcard)
      - Semantics: compile-time "unconstrained" — the actual type is determined by usage
      </pros>
      <cons>Overloads `*` which already means collect operators. But context is unambiguous (`;*` is always type position).</cons>
    </option>

    <option id="gap-3">
      <name>Gap 3: Enum vs Value field kind</name>
      <pros>
      Structs have enum fields (no `;type`) and value fields (with `;type`). Is this implicit from declaration or does it need a `%Kind` property?

      **Recommendation: Implicit — no `%Kind` property needed.**
      - Field kind is syntactically obvious: `[.] .name` (enum) vs `[.] .name;type` (value)
      - The compiler already knows from parsing — no need to restate in metadata
      - Adding `%Kind` adds overhead with zero information gain
      </pros>
      <cons>Less explicit in the metadata tree — you'd need to check the definition source to know field kind.</cons>
    </option>

    <option id="gap-4">
      <name>Gap 4: Key uniqueness property</name>
      <pros>
      Keys are always unique in Polyglot (tree semantics — a node can't have two children with the same name).

      **Recommendation: No `%Key.Unique` property — uniqueness is a tree invariant.**
      - All Polyglot data is trees → keys are always unique at any given level
      - This is not a property of collections, it's a property of the data model
      - No user-defined collection can violate this (multisets are not trees)
      </pros>
      <cons>If anyone ever wants a multiset/bag type, they'd need a different representation (e.g., value-based counting).</cons>
    </option>

    <option id="gap-5">
      <name>Gap 5: `[#]` overloading</name>
      <pros>
      `[#]` currently does: (a) data loading, (b) type parameter declaration, (c) schema property declaration.

      **Recommendation: Keep `[#]` — it's all "definition metadata."**
      - `[#]` means "this is definition-level information" in a `{#}` block
      - Data loading (`[#] .field << value`), type params (`[#] <param`), and schema props (`[#] %Prop`) all configure the definition
      - The prefix after `[#]` disambiguates: `.` = field default, `<` = type param, `%` = schema property
      - Adding `[%]` for a single use case (schema props) adds a new block marker for no semantic benefit
      </pros>
      <cons>Three different syntactic forms under one marker. Users need to learn that `[#]` context depends on what follows it.</cons>
    </option>

    <option id="gap-6">
      <name>Gap 6: User-defined collection usage syntax</name>
      <pros>
      If a user defines `{#} #SortedSet<type` with `[#] %Ordered << #True`, how is it used?

      **Recommendation: Usage follows the collection's field kind.**
      - If the collection uses `[:]` flexible fields → usage is `;SortedSet:int` (flex separator)
      - If it uses `[.]` fixed fields → usage is `;SortedSet.int` (fixed separator)
      - This is consistent with how `;array.int` (was fixed `.`) and `;dict:string:int` (flexible `:`) already work
      - **With the array `.` → `:` change**, arrays become `;array:int` too — making ALL collections use `:` for params
      </pros>
      <cons>After the array migration, the `.` separator in type annotations becomes rare (only for struct field access, not collection params).</cons>
    </option>

    <option id="gap-7">
      <name>Gap 7: Array `.` to `:` migration scope</name>
      <pros>
      Changing array keys from fixed (`.0`, `.1`) to flexible (`:0`, `:1`) touches:
      - `docs/user/syntax/types.md` — array type annotation section
      - `docs/user/concepts/collections.md` — collection table, examples
      - `docs/technical/EBNF.md` — `array_type` production (`fixed_sep` → `flex_sep`)
      - `docs/technical/EDGE-CASES.md` — EC references to `array` with `fixed_sep`
      - `docs/technical/compile-rules/PGE/PGE04013-nested-array-type.md` — nested array rule
      - `docs/technical/compile-rules/PGE/PGE04017-array-dimension-access-mismatch.md` — dimension access
      - Any examples using `$arr.0` syntax

      **Recommendation: Migrate in Plan 02 as part of spec updates.**
      - Touch each file once with all changes (array `:` + schema properties + dict)
      - Grep for `$arr.0`, `array.*fixed`, `.0;`, `[.].*array` patterns
      </pros>
      <cons>Large blast radius. But these files all need updating anyway for schema properties.</cons>
    </option>

  </options>
  <resume-signal>Review each gap recommendation. For each, reply "approved" or provide your preferred resolution. You can approve all at once with "all approved".</resume-signal>
