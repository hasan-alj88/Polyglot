    Create 4 individual PGE rule files following the exact format of PGE-912-duplicate-import-alias.md:
    - YAML frontmatter: rule, code, name, severity
    - Statement, Rationale, Detection, See also
    - VALID and INVALID code examples using `[ ] ✓`, `[ ] ✗ PGE-NNN` comment markers

    **PGE-917 — Unknown Permission Category** (Rule 9.17)
    - Statement: A `[_]` declaration must use a known permission category and subfield. If the category or subfield is not in the predefined set, PGE-917 fires.
    - Rationale: Permission categories are Polyglot-defined (not user-extensible). Unknown categories would create permissions that no IO operation can consume.
    - Detection: The compiler checks `_Category.subfield` against the 8 known categories (File, Web, Database, System, Crypto, IPC, Device, Memory) and their known subfields.
    - VALID: `[_] _File.read"/var/log/*"` — known category and subfield
    - INVALID: `[_] _Network.send"*"` — unknown category "Network"; `[_] _File.rename"/tmp/*"` — unknown subfield "rename"
    - See also: PGE-918 (undeclared permission), [[permissions#Permission Categories]]

    **PGE-918 — Undeclared Permission** (Rule 9.18)
    - Statement: If a pipeline calls a stdlib IO pipeline (e.g., `=File.Text.Read`) without a matching `[_]` declaration, PGE-918 fires. The compiler traces the call graph to verify.
    - Rationale: Implicit-deny model — every IO operation must be explicitly permitted. Call graph tracing catches both direct IO calls and transitive calls through other pipelines.
    - Detection: The compiler builds a call graph from each pipeline's execution body. For every call to a known IO pipeline, it checks that the calling pipeline has a matching `[_]` declaration. This includes transitive calls: if `=A` calls `=B` which calls `=File.Text.Read`, then `=A` must have `[_] _File.read` covering the path.
    - VALID: Pipeline with `[_] _File.read"/var/log/*"` calling `=File.Text.Read >> "/var/log/app.log"`
    - INVALID: Pipeline calling `=File.Text.Read` without any `[_] _File.read` declaration
    - Use the issue's code example for the INVALID case
    - See also: PGE-915 (exceeds ceiling), PGE-917 (unknown category), [[permissions#Compile-Time Enforcement]]

    **PGE-919 — Permission Output** (Rule 9.19)
    - Statement: Permission IO declarations (`[_]`) are input-only. Any `[_] >param` (output direction) is a compile error.
    - Rationale: Permissions declare what IO is allowed, not what is returned. Output direction is meaningless in a permission context.
    - Detection: The compiler checks all `[_]` IO lines. If any line uses `>` (output direction) instead of `<` (input direction), PGE-919 fires.
    - VALID: `[_] <url;string << "https://api.example.com/*"` — input direction
    - INVALID: `[_] >result;string` — output direction in permission block
    - See also: PGE-810 (IO direction mismatch), [[permissions#IO Form]]

    **PGE-920 — Duplicate Permission** (Rule 9.20)
    - Statement: The same permission cannot be declared twice in one `[_]` block (whether in `{@}` ceiling or `{=}` pipeline). If the same category.subfield appears more than once, PGE-920 fires.
    - Rationale: Duplicate permissions are ambiguous — which scope wins? Even if identical, duplicates indicate copy-paste errors or incomplete refactoring.
    - Detection: The compiler collects all `[_]` declarations within each block scope (`{@}` or `{=}`). If the same `_Category.subfield` appears more than once, PGE-920 fires on the second occurrence.
    - VALID: Pipeline with `[_] _File.read"/var/log/*"` and `[_] _File.write"/tmp/*"` — different subfields
    - INVALID: Pipeline with two `[_] _File.read` lines (even with different paths — the capability is the same, paths could use globs)
    - See also: PGE-912 (duplicate import alias — analogous), PGE-915 (ceiling check)

    **PGW-903 — Unused Permission** (Rule 9.3)
    - Statement: A `[_]` permission declared in a pipeline or macro that is never exercised by any IO call in its call graph is flagged with a warning.
    - Rationale: Unused permissions are dead declarations — they grant IO capability that is never used. This may indicate incomplete refactoring (an IO call was removed but the permission was left behind) or over-broad permission requests.
    - Detection: The compiler traces the call graph from the definition. For each `[_]` declaration, it checks whether any call in the graph exercises that permission. If no call matches, PGW-903 fires.
    - VALID: Pipeline with `[_] _File.read"/var/log/*"` that calls `=File.Text.Read >> "/var/log/app.log"` — permission exercised
    - WARNING: Pipeline with `[_] _File.read"/var/log/*"` and `[_] _Web.request` but only calls `=File.Text.Read` — `_Web.request` is unused
    - See also: PGW-902 (unused import — analogous pattern), PGE-918 (undeclared permission — the inverse)
