<objective>
## Goal
Resolve two EBNF edge cases from issue #304:
- X.33: Fallback operators (`!<`/`!>`) used where the source expression is not a pipeline call (non-failable context)
- X.34: `[=]`/`[b]` parallelism markers must pair with a next `[=]`/`[b]` sibling; mismatch is a compile error

## Purpose
Tighten the compiler's semantic rules so that meaningless operator combinations are caught at compile time, preventing developer confusion.

## Output
- 3 new compile rules: PGE07008 (fallback on non-failable source), PGE07009 (unterminated fallback chain), PGE01040 (orphan parallel marker)
- Updated EBNF prose clarifications in §6.1 and §12.2
- New edge cases: X.33 in 06-operators.md, X.34 in 12-collections.md
- User-facing doc updates for operators and collectors
</objective>

## AC-1: PGE07008 — Fallback on non-failable source
```gherkin
Given a fallback operator (!< or !>) is used in any assignment context
When the source expression (RHS) is a literal value or variable reference (not a pipeline call)
Then PGE07008 fires — fallback operators only activate on error; a literal or variable cannot error
```

## AC-2: PGE07009 — Unterminated fallback chain
```gherkin
Given a fallback chain (!< -Pipeline1 !< -Pipeline2 !< ...)
When the chain does not terminate at a non-failable expression (literal or variable)
Then PGE07009 fires — fallback chain must terminate at a value that cannot fail
```

## AC-3: PGE01040 — Orphan parallel marker
```gherkin
Given a line uses [=] or [b] execution marker
When the next sibling line does not also use [=] or [b]
Then PGE01040 fires — parallel marker requires a parallel sibling to pair with
```

  <action>
    **PGE07008 — Fallback on Non-Failable Source:**
    - `!<`/`!>` is only valid when the source expression is a pipeline call (which can fail)
    - If source is a literal or variable reference, PGE07008 fires
    - This applies universally — any production using `assignment_op` where the RHS is non-failable
    - Examples: schema_property, expand_io_line, collect_io_line input, value_field, metadata — all error when RHS is literal/variable
    - Exception: collect_io_line OUTPUT (`(*) >>`) is valid with fallback because collection processes can fail (timeouts, all-Failed)
    - Exception: call_io_line, chain_io_line — pipeline calls are failable
    - Exception: data_load with inline pipeline — pipeline can fail, fallback is valid
    - See also: PGE07003 (duplicate fallback), PGW07004 (fallback on non-failable IO)

    **PGE07009 — Unterminated Fallback Chain:**
    - A fallback chain (`$x !< -Pipeline1 !< -Pipeline2 !< ...`) must terminate at a non-failable expression
    - The terminal must be a literal value or variable reference, not another pipeline call
    - If the chain ends at a pipeline call, PGE07009 fires — there is no final recovery value
    - The compiler walks the chain left-to-right; every `!<`/`!>` that has a pipeline RHS must itself have a fallback
    - See also: PGE07008 (the general failable-source rule)

    **PGE01040 — Orphan Parallel Marker:**
    - `[=]` or `[b]` means "run in parallel with the next `[=]` or `[b]` sibling"
    - If the next sibling at the same indentation level is NOT `[=]` or `[b]`, PGE01040 fires
    - `[*]`, `[-]`, and other markers are inherently sequential — they cannot be paired with `[=]`/`[b]`
    - This replaces the vague "lone collector with [=]" concern from the issue
    - See also: PGE01024 (incompatible operation marker), PGE03005 ([b] has no collectible output)
