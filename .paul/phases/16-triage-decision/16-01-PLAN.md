<tasks>

<task type="auto">
  <name>Task 1: Build triage report</name>
  <files>All 52 files in docs/archive/</files>
  <action>
    Read every archived file. For each, assess:
    1. Cross-reference density (from Phase 15 enrichment):
       - High (4-5 refs): content has strong links to current spec
       - Medium (2-3 refs): some links, may have extractable value
       - Low (1 ref): minimal connection, likely just @d: deprecation notice
    2. replaced_by: field — does a current-spec file fully cover this content?
       - "none" = no replacement exists → higher keep/extract likelihood
       - specific file = fully superseded → lean toward delete
    3. Content uniqueness — does this file contain anything NOT in the replacement?
       - Historical context, design rationale, examples, edge cases

    Produce a triage table grouped by category:
    | File | Refs | replaced_by | Verdict | Rationale |
    
    Triage rules (guidelines, not absolute):
    - replaced_by: specific file + low refs → DELETE (fully superseded)
    - replaced_by: none → KEEP or EXTRACT (no replacement exists)
    - High refs + unique content → EXTRACT (pull useful bits) or KEEP
    - Audit/process docs with only @d: → DELETE (no reusable content)
    - Index/README/SUMMARY docs → DELETE (structural, not content)

    Present the report to user — do NOT execute any changes yet.
  </action>
  <verify>Report covers all 52 files with verdict and rationale for each</verify>
  <done>AC-1 satisfied: every file triaged with verdict and rationale</done>
</task>

<task type="checkpoint:decision" gate="blocking">
  <decision>User must approve the triage verdicts before any files are modified or deleted</decision>
  <context>
    This is a destructive operation — deleted files cannot be recovered without git history.
    The user may want to adjust individual verdicts (e.g., keep a file marked for deletion).
  </context>
  <options>
    <option id="approve-all">
      <name>Approve all verdicts as-is</name>
      <pros>Fast, trust the analysis</pros>
      <cons>No manual review of edge cases</cons>
    </option>
    <option id="modify">
      <name>Modify specific verdicts</name>
      <pros>Fine-grained control over what stays/goes</pros>
      <cons>Takes more time</cons>
    </option>
  </options>
  <resume-signal>Type "approved" to execute all verdicts, or list files to change (e.g., "keep architecture.md, delete draft.md")</resume-signal>
</task>

<task type="auto">
  <name>Task 3: Execute approved triage decisions</name>
  <files>docs/archive/ (files marked DELETE)</files>
  <action>
    For each approved verdict:
    - DELETE: Remove the file with git rm
    - EXTRACT: Add a note to the file's frontmatter with extract_sections field listing what to pull (actual extraction is future work, not this plan)
    - KEEP: No action needed

    After all deletions:
    - Remove any empty directories left behind
    - Do NOT commit — leave staged for user to review with git diff --cached
  </action>
  <verify>git status shows only approved deletions staged; no files outside docs/archive/ affected</verify>
  <done>AC-3 and AC-4 satisfied: approved deletions executed, current spec untouched</done>
</task>
