    Create the cybersecurity philosophy page following the established format from accountability.md:

    **Frontmatter:** audience: [automation-builder, integrator, architect, designer], type: reference, updated: 2026-04-20

    **Opening:** `<!-- @c:vision -->` import, then H1 "Cybersecurity", then blockquote connecting to [[vision]] context.

    **Sections to write (philosophy-level, not technical spec):**

    1. **Zero Trust on Executed Code** — Polyglot's runtime enforces implicit deny independently of the compiler. Even if code somehow bypasses compile-time checks, the Polyglot Service rejects unauthorized actions. This is system-enforced, not honor-system. Reference [[user/concepts/permissions/enforcement]] for technical details. Connect to core-philosophy.md's "Implicit Deny Permission" principle.

    2. **No Permission Inheritance** — Permissions are never inherited, never passed down. Every package, pipeline, and task declares its own permissions independently. The only exception is the package ceiling ({@}), which sets an upper bound — but a ceiling is a limit, not a grant. No object can piggyback on another's permissions. Reference [[user/concepts/permissions/hierarchical-scoping]] for scoping rules.

    3. **Black Box Monitoring Is Mandatory** — Polyglot cannot analyse what it cannot see. Compiled binaries and AST-invisible constructs are opaque — the compiler cannot verify their internal behaviour. So Polyglot watches what they *do*: OTel events track permission checks, resource usage, error patterns, and anomalous behaviour at runtime. This is not optional telemetry — it is mandatory security infrastructure. Reference [[technical/spec/otel-permission-events]] for the 8 security events.

    4. **Black Box Trust Metric** — All black boxes (compiled binaries via -Run.*, code using AST-invisible functions from the banned registry) are tracked. Metrics include: permission breach attempts, error rates, resource consumption patterns, anomalous behaviour. These metrics feed into the cybersecurity section of the Polyglot audit report. A black box that misbehaves builds a trust deficit that operators can act on. Reference [[technical/compiler/ast-invisible-registry]] for what constitutes a black box.

    **Closing:** "Related Philosophy" section with wikilinks to core-philosophy, language-design, accountability, and error-philosophy (planned — #335).

    **Style guidance:**
    - Match accountability.md tone: direct, declarative, principle-first
    - Use [[wikilinks]] for Obsidian cross-references
    - Explain *why* these rules exist, not *how* they are implemented
    - Do not duplicate PGE numbers, EBNF grammar, or sandbox setup steps
    - Keep under 50KB (this should be well under)
