<objective>
## Goal
Replace the old `<InlineStringLiteral` mechanism with the new `%InlineString` template-based inline pipeline invocation syntax. Fix stale `=Pipeline` references. Add compiler rules for template validation. Update pglib pipelines to use the new syntax.

## Purpose
The inline pipeline call syntax `-Pipeline"string"` currently uses an opaque `<InlineStringLiteral#string` parameter — the pipeline receives one raw string and must parse it internally. The new `%InlineString` template design lets pipelines declare a structured template (e.g., `"{host}:{port}/{db}"`) where placeholder names map directly to declared `(-)` inputs. The compiler extracts arguments and wires them automatically, enabling compile-time format validation and type coercion.

Also fixes stale `=Path` / `=Pipeline` references left from before the `=` prefix was reserved exclusively for expanders.

## Design Decisions (from discussion)
- `(-) %InlineString << "{template}"` declares the inline template in the IO section
- `{name}` = required placeholder, must match a declared `<name` input
- `{name?}` = optional placeholder, matched input MUST have `<~` default
- Compiler extracts args from string, wires to inputs by name matching
- `<InlineStringLiteral` is fully retired — replaced by %InlineString
- `.inlineFormat` / `.inlineExamples` metadata replaced by the template itself
- Dual-mode pipelines work: inline call uses template extraction, normal call ignores %InlineString
- All `=Pipeline"..."` references must be fixed to `-Pipeline"..."`
