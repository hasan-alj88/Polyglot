<objective>
## Goal
Document job hierarchy addressing syntax in Polyglot Code: nested `[Q]` scoping under markers, `=Q.Job.Branch` aj3lib pipeline, positional job paths in the metadata tree, and EBNF grammar updates.

## Purpose
Issue #115 — Polyglot pipelines break into jobs at IO boundaries and jobs form a tree. The architecture (Queue Manager, TM, NoSQL schema) already supports this, but there is no user-facing syntax for addressing individual jobs or branches, and no grammar for nested `[Q]` under markers. This plan formalizes the design decisions made during discussion.

## Output
- Updated user docs: queue.md (nested [Q] under markers), Q.md (=Q.Job.Branch aj3lib pipeline)
- Updated spec docs: branches.md (positional job paths), path-grammar.md (job path grammar)
- Updated EBNF: 09-definition-blocks.md (nested [Q] in execution section)
</objective>

    ```ebnf
    job_path ::= instance_path ".jobs" ":" uid { "." field }
               | instance_path "." marker_type "." index ;
    marker_type ::= "r" | "p" | "b" ;
    ```
    - Add a row to the element table explaining `marker_type` and `index`
    - Note: `job_path` via `marker_type.index` is compiler-internal only; user code accesses jobs through `.jobs:UID`
  </action>

    In the execution_section or call_expr production (§10 or §8), add an optional `queue_control_line` after a call expression's IO lines. The pattern:

    ```ebnf
    call_expr ::= indent marker pipeline_ref NEWLINE

    ```

    Where `queue_control_line` is already defined in §9.3.3. This allows:


    Add a note: "Job-level `[Q]` extends pipeline-level queue conditions for this specific job. See [[concepts/pipelines/queue#Job-Level Queue Conditions]]."

    Avoid: Changing the pipeline_body production — pipeline-level [Q] stays in the header. Only execution-body call expressions get nested [Q].
  </action>
