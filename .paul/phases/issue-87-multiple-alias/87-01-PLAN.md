<objective>
## Goal
Rename `#IndexString` to `#KeyString`, introduce `#NestedKeyString` (allows `.` and `:`), convert `.alias` from a singular fixed metadata field to a flexible `%alias#Array.NestedKeyString` field with `[:]` assignment, update PGE01015 to remove `.alias` from the fixed-field list, and add PGE12002 for globally unique alias enforcement.

## Purpose
Enables definitions to have multiple shorthand aliases (e.g., `!Permission.File.Denied` aliased as both `!File.Permission.Denied` and `!FileDenied`), which was blocked by the single-alias design.

## Output
- `#IndexString` renamed to `#KeyString` across all spec files
- New `#NestedKeyString` type definition (allows `.` and `:`)
- `.alias` converted to flexible `%alias` with array semantics
- PGE01015 updated (`.alias` removed from fixed-field duplicate check)
- PGE12002 created (duplicate alias — fully resolved name must be globally unique)
- EBNF `metadata_alias` production updated
- Edge cases updated
</objective>

    2. Create PGE12002-duplicate-alias.md:
       - Rule: 10.2
       - Code: PGE12002
       - Name: Duplicate Alias
       - Severity: error
       - Statement: Each alias string within a single %alias declaration must be unique, AND each fully resolved alias name must be globally unique across all definitions. Duplicate alias names are a compile error.
       - Rationale: Aliases create alternative lookup paths. If two definitions resolve to the same alias, the compiler cannot determine which definition to use. This includes nested paths — `!File.Permission.Denied` declared as an alias from two different error branches is ambiguous.
       - Detection: Compiler collects all alias strings from all %alias declarations, resolves them to fully qualified paths, and rejects any duplicates.
       - Include VALID and INVALID examples
       - See also: PGE01015, PGE09011 (duplicate import alias)
    3. In COMPILE-RULES.md:
       - Add PGE12002 to the rules table in the appropriate section (10.x metadata rules)
