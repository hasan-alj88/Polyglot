<objective>
## Goal
Document the full -T.Call signal path: how the SDK's `call()` function reaches the Trigger Monitor via NATS, how TM processes the request, enqueues the job, collects results, and returns them to the SDK. Focus on the TM-side signal processing.

## Purpose
This closes the gap between the SDK spec (#319) and the existing QH signal architecture (#265). The SDK defines `call()` with NATS topics, and the QH docs define `command.enqueue` — but nothing documents how the TM bridges these two: receiving SDK calls, matching -T.Call pipelines, writing bindings to the NoSQL store, and publishing results back.

  <name>Task 1: Add -T.Call Signal Path section to native-dispatch.md</name>
  <files>docs/technical/spec/native-dispatch.md</files>
  <action>
    Add a new section "## -T.Call Signal Path" after the existing "### Trigger" subsection under "#NativeKind Routing". This is the core technical content.

    Content to include:

    1. **Overview** — -T.Call is unique among triggers: it uses NATS request-reply (synchronous from SDK's perspective) rather than fire-and-forget event evaluation. The TM acts as a bridge between the SDK's NATS call and the standard command.enqueue pipeline.

    2. **TM Subscription** — At startup, TM subscribes to `polyglot.call.*` wildcard. When a message arrives on `polyglot.call.{pipeline_name}`, TM extracts the pipeline name from the subject.

    3. **Pipeline Matching** — TM searches its trigger registry for a pipeline with `[T] -T.Call` matching the requested pipeline name. If not found, TM publishes an error response to `polyglot.result.{correlation_id}` with `!SDK.PipelineNotFound`.

    4. **Binding Storage** — TM writes SDK-provided input bindings to the NoSQL variable store using the existing Redis key pattern:
       - `job:{UID}:port:{port_name}` — one key per input binding
       - Each value is the typed JSON envelope from the SDK request payload
       - The UID is generated by TM at this point (not by the SDK)
       - TM also stores correlation metadata: `job:{UID}:meta:correlation_id` and `job:{UID}:meta:caller_topic`

    5. **Enqueue** — TM emits `polyglot.command.enqueue` with the standard signal payload (jobId, pipeline, queue, params). The job flows through QH → Runner as normal.

    6. **Result Collection** — TM subscribes to `polyglot.runner.completed.{jobId}` and `polyglot.runner.failed.{jobId}`. On completion:
       - Success: TM reads output ports from `job:{UID}:port:{name}` (output direction), constructs the success response envelope, publishes to `polyglot.result.{correlation_id}`
       - Failure: TM reads the error from the runner.failed signal, constructs the error response envelope, publishes to `polyglot.result.{correlation_id}`

    7. **Sequence Diagram** — Text-based sequence diagram showing:
       SDK → NATS(polyglot.call.X) → TM → Redis(write bindings) → NATS(command.enqueue) → QH → NATS(control.start) → Runner → NATS(runner.completed) → TM → Redis(read outputs) → NATS(polyglot.result.UUID) → SDK

    8. **Comparison Table** — How -T.Call differs from -T.Daily, -T.Webhook etc:
       | Aspect | -T.Call | Other Triggers |
       | Signal source | SDK via NATS | TM evaluation loop |
       | Response | Synchronous (request-reply) | None (fire-and-forget) |
       | Bindings | SDK provides inputs | Trigger provides outputs |
       | Correlation | Per-call UUID | None |

    Update the `updated:` frontmatter date to 2026-04-19.
