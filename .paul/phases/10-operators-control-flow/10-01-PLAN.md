<task type="auto">
  <name>Task 2: Draft conditionals.md — exhaustiveness, logical operators, nesting</name>
  <files>docs/user/concepts/conditionals.md</files>
  <action>
    Create new spec file `docs/user/concepts/conditionals.md` covering all conditional/control-flow semantics. Structure:

    **Frontmatter:** audience: user, type: specification, updated: 2026-03-24, status: complete

    **Smart references:** @operators, @blocks, @types, @variable-lifecycle

    **Sections:**

    1. **Conditional Chains (`[?]`)** — basic structure:
       - Sequential `[?]` blocks act as switch-case (not if/else-if)
       - Each branch contains execution lines indented under it
       - Cross-ref [[blocks#Control Flow]] for marker reference
       - Example from EC-11.1

    2. **Comparison Operators in Conditionals:**
       - Brief note: "See [[operators#Comparison Operators]] for the full operator table"
       - Key rule: every [?] arm MUST include a comparison operator — bare subject lines are invalid (PGE06009)
       - Empty conditional scope is invalid (PGE06010)

    3. **Exhaustiveness Rules** — the core of this spec:
       - **Enum exhaustiveness:** If all enum variants are covered, no `*?` needed (PGE06002). Example from EC-22.1.
       - **Numeric range exhaustiveness:** Ranges must cover the full domain or include `*?` (PGE06003, PGE06004)
       - **String exhaustiveness:** Strings are open sets — always need `*?` (PGE06006)
       - **Flexible field exhaustiveness:** Open keys — always need `*?` (PGE06007)
       - **Compound condition exhaustiveness:** AND/OR combinations — cross-ref algorithm (PGE06008)
       - **Duplicate wildcard:** Only one `*?` per chain (PGE06011)
       - **Unreachable after wildcard:** Branches after `*?` are dead code (PGE06012)
       - **Tautological branch:** Always-true condition is suspicious (PGE06013)
       - **Overlap detection:** Ranges/compounds that overlap — cross-ref PGE06004, PGE06005

    4. **Logical Operators** — compound conditions:
       - `[&]` AND — both conditions must hold
       - `[+]` OR — at least one condition holds
       - `[-]` NOT — negate condition (note: prefer negation operators `=!?` etc. for simple negation)
       - `[^]` XOR — exactly one condition holds
       - Examples from EC-11.3 and EC-22.4
       - Cross-ref [[blocks#Logical]] for marker definition

    5. **Nested Conditionals:**
       - A `[?]` branch can contain inner `[?]` chains
       - Each nesting level is independently exhaustive
       - Example from EC-22.2

    6. **Switching on Metadata:**
       - `[?]` can switch on `%status` or other live metadata fields
       - Inner branches check enum variants
       - Example from EC-22.3

    Use [[wikilinks]] throughout. Follow docs/audit/ conventions. Keep examples concise.
    Do NOT duplicate the operator table from operators.md — cross-reference it.
  </action>
