<objective>
## Goal
Tighten the `exec_expr` EBNF production to reject bare literals and non-pipeline identifiers, and add a compile error rule (PGE01020) for effectless execution expressions.

## Purpose
The EBNF grammar currently allows `exec_expr ::= ... | identifier`, which lets bare identifiers (`#UserRecord`, `$existingVar`, `@AD`) and bare literals (`42`, `"string"`) parse as valid execution expressions under `[r]`, `[p]`, `[b]` markers. These produce no effect — they aren't assignments, pipeline calls, chains, or expand invocations. This is a grammar permissiveness bug. Resolves GitHub issue #97.

## Output
- Tightened `exec_expr` EBNF (remove `| identifier` catch-all)
- New compile rule PGE01020 with detail file
- New edge cases EC-10.12 and EC-10.13
- Updated indexes
</objective>

    2. Create compile-rules/PGE/PGE01020-effectless-execution-expression.md:
       - Rule 1.20, code PGE01020, name "Effectless Execution Expression", severity error
       - Statement: An execution expression under [r], [p], or [b] must be an assignment, pipeline call, chain call, or expand invocation. Bare literals and non-pipeline identifiers produce no effect and are compile errors.
       - Rationale: Execution markers define how work runs. A bare literal or identifier is not work — it assigns nothing, calls nothing, and produces no side effect. Allowing it is misleading dead code.
       - VALID examples: `[r] =Transform ...`, `[r] $result <~ "value"`, `[p] =Parallel.Work ...`
       - INVALID examples (PGE01020): `[r] 42`, `[r] "orphaned string"`, `[r] #UserRecord`, `[p] $existingVar`, `[b] @AD`
       - See also: PGE01016 (unmarked execution line), PGW02002 (unused variable)
       - Follow existing rule file format (frontmatter with rule/code/name/severity, then content)

       - After PGE01019 row, add: `| PGE01020 | Effectless Execution Expression |`

    Avoid: Do not change any other exec_expr alternatives or other EBNF productions. Only remove `| identifier`.
