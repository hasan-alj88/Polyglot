    Create 4 PGE error rule files following the exact format of PGE10010:

    **PGE10011 — Shell Without Capability (Rule 9.25):**
    - Statement: -Run.Shell pipeline without {_} #System.#Shell capability is compile error
    - Detection: compiler checks any {-} using -Run.Shell has (-) _PermName where {_} has .category #System and .capability #Shell
    - VALID: -Run.Shell with matching {_} #System.#Shell
    - INVALID: -Run.Shell with no system permission, or with wrong capability
    - Cross-refs: @u:concepts/permissions/foreign-code, @u:pglib/pipelines/Run/Shell

    **PGE10012 — Code File Outside Scope (Rule 9.26):**
    - Statement: <code.file path outside any declared {_} .scope is compile error
    - Detection: resolve <code.file path literal, check against all declared {_} .scope patterns with .category #File
    - VALID: <code.file "/scripts/etl.py" with {_} .scope "/scripts/*"
    - INVALID: <code.file "/tmp/rogue.py" with {_} .scope "/scripts/*"
    - Cross-refs: @u:concepts/permissions/foreign-code, @u:concepts/permissions/enforcement

    **PGE10013 — Foreign Resource Outside Scope (Rule 9.27):**
    - Statement: foreign code literal path/host/connection outside declared {_} scope is compile error
    - Detection: AST walk finds IO call in sink table, extracts resource argument (string literal or traceable variable), glob-matches against {_} .scope
    - VALID: pd.read_csv("/data/reports/q1.csv") with {_} .scope "/data/reports/*"
    - INVALID: open("/etc/shadow") with {_} .scope "/data/reports/*"
    - Also show network example (requests.get URL outside .host)
    - Cross-refs: @u:concepts/permissions/foreign-code, @c:technical/algorithms/foreign-code-analysis (plan 311-02)

    **PGE10014 — AST-Invisible Foreign Code (Rule 9.28):**
    - Statement: eval/exec/importlib/ctypes/dlopen/inline asm in foreign code is compile error
    - Detection: AST walk checks each node against BANNED_CONSTRUCTS per language
    - Include the per-language banned constructs table from brainstorm
    - VALID: standard library calls (open, read, write)
    - INVALID: eval("open('/etc/passwd')"), exec(user_input), __import__('os')
    - Cross-refs: @u:concepts/permissions/foreign-code

    All files use:
    - audience: developer
    - severity: error
    - Rule numbering continues from 9.24 (PGE10010)
    - All Polyglot code examples use the current {_} permission-as-resource syntax with .intent, .category, .capability, .scope, .path fields
    - All pipeline references use -Run.* naming (not -RT.*)
  </action>

  <name>Task 2: Create 4 PGW warning rule files (PGW10002/10003/10005/10006)</name>
  <files>

  <action>
    Create 4 PGW warning rule files following the exact format of PGW10001:

    **PGW10002 — Unverifiable Foreign IO (Rule 9.4):**
    - Statement: IO call detected in foreign code but resource value not fully resolvable — compiler cannot verify scope compliance
    - Detection: AST finds sink-table call, extract_argument returns Variable that trace_assignment cannot resolve to a literal (function return, external input, cross-function value)
    - VALID (no warning): pd.read_csv("/data/q1.csv") — literal, fully verifiable
    - WARNING: path = get_path(); pd.read_csv(path) — traces to function call, unresolvable
    - Also show: resource from <Bind input that traces to known Polyglot IO type — medium confidence, still warns
    - Cross-refs: @u:concepts/permissions/foreign-code, @c:technical/algorithms/foreign-code-analysis

    **PGW10003 — Bind Mode Opacity (Rule 9.5):**
    - Statement: -Run.*.Bind pipelines are fully opaque — compiler cannot analyze foreign code that pulls/pushes via API
    - Detection: always fires on any {-} using -Run.*.Bind mode
    - WARNING: any -Run.Python.Bind pipeline — "foreign code controls data flow via pull()/push(), compiler cannot verify permission compliance; sandbox-only enforcement"
    - Cross-refs: @u:concepts/permissions/foreign-code, @u:pglib/pipelines/Run/Bind

    **PGW10005 — Unrecognized Foreign Call (Rule 9.6):**
    - Statement: function call in foreign code not found in IO registry — compiler cannot classify as IO or pure
    - Detection: AST finds CallExpression, canonical name not in sink table. Bottom-up: if function body contains known IO calls, flag with the IO category. If truly unknown, warn.
    - WARNING: custom_lib.process(data) — not in sink table, no known IO subcalls
    - Note: known-pure builtins (len, str, sorted, etc.) are silently skipped
    - Cross-refs: @u:concepts/permissions/foreign-code, @c:technical/compiler/io-registry

    **PGW10006 — Shell Variable Expansion (Rule 9.7):**
    - Statement: shell $VAR or ${VAR} in IO context within -Run.Shell — compiler cannot verify expanded value
    - Detection: shell AST walk finds variable expansion nodes within command arguments that map to IO operations (file paths, URLs)
    - WARNING: echo "$LOG_PATH/output.log" in IO context — variable expansion prevents scope verification
    - VALID (no warning): echo "hello" — no variable expansion in IO context
    - Cross-refs: @u:concepts/permissions/foreign-code, @u:pglib/pipelines/Run/Shell

    All files use:
    - audience: developer
    - severity: warning
    - Warning numbering continues from 9.3 (PGW10001)
    - All examples use -Run.* naming and current {_} syntax
  </action>
