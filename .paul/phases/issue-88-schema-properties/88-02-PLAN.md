<objective>
## Goal
Update collections.md to reflect the new collection hierarchy: #Map as universal base, #Array as #Map variant, #Serial as schema-free tree, user-defined structs, and the idiomatic dataframe pattern.

## Purpose
Issue #88 redesigns collections as tree structures with declarative schema properties. #Map replaces #Dict as the base collection, #Array becomes a #Map variant with Cartesian integer keys, and #Serial drops all schema constraints. The collection spec must document these changes with the approved {#} definitions.

## Output
- Updated `docs/user/concepts/collections.md` with new collection hierarchy, definitions, and patterns
</objective>

## AC-2: #Array as #Map Variant
```gherkin
Given #Array inherits from #Map with #UnsignedInt keys
When the plan is applied
Then collections.md contains the {#} #Array<ValueType<Dim definition from draft.md
And documents <~ #Map inheritance with ##Contiguous + ##Rectangular
And documents <Dim#Dimension << "1D" default parameter syntax
And shows Cartesian product key structure for multidimensional arrays
And shows access: $myArray<0, $matrix<0<1 (2D)
```

## AC-3: #Serial as Schema-Free Tree
```gherkin
Given #Serial has no schema constraints — accepts any compilable tree
When the plan is applied
Then collections.md contains the {#} #Serial definition with only %##Alias and [:] :*#*
And documents that no ## schemas are applied — fully unconstrained
```

## AC-4: User-Defined Struct Pattern
```gherkin
Given user structs use [.] fixed fields with . accessor
When the plan is applied
Then collections.md documents the {#} #Person example
And documents that ## is optional — compiler infers from field declarations
And documents . accessor for fixed fields vs < for flexible children
```

## AC-5: Idiomatic Dataframe Pattern
```gherkin
Given #Dataframe is dropped as built-in — row struct + #Array is idiomatic
When the plan is applied
Then collections.md documents the #SalesRow + #SalesTable pattern from draft.md
And documents access: $sales<0.product (mixed < and . accessors)
And documents that tables are built via *Into.Array collectors
```

## AC-6: Nested Collection Depth Safety
```gherkin
Given Decision #21 requires bounded nesting
When the plan is applied
Then collections.md documents that collections used as value types require explicit %##Depth.Max
And references PGE-922 for unbounded nesting and PGW-906 for unlimited depth on user types
```
