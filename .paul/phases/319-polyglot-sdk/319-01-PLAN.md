<objective>
## Goal
Create the Polyglot SDK specification document covering cross-language type conversion, call protocol, wire format, and distribution.

## Purpose
Polyglot orchestrates foreign code in separate processes. The SDK is the formal contract for how external applications serialize data, trigger pipelines, and exchange values during execution. Without this spec, each language integration would be ad-hoc.

New file: `docs/technical/spec/polyglot-sdk.md`

       - `to_polyglot(value, type) → str` — serialize native → Polyglot string
       - `from_polyglot(str) → value` — deserialize Polyglot string → native
       - `call(pipeline, bindings) → result` — trigger pipeline via -T.Call (fire-and-wait). NATS request-reply on polyglot.call.* / polyglot.result.* topics.
       - `pull(name) → value` / `push(name, value)` — Bind protocol for ongoing data exchange inside -Run.*.Bind jobs

       | Polyglot | Python | Rust | Go | JavaScript |
       Escape pipes in cells as needed.
       State: "Bindings use only primitive types — no custom or user-defined types."

    5. **Type Mapping Descriptor** — Per-IO-port descriptor emitted by compiler in Behavior Contract:
       - Schema: { port_name, direction (input/output), polyglot_type, is_array, element_type }
       - Purpose: Runner/SDK serialize without runtime type inspection
       - Show example descriptor for a pipeline with mixed IO types
       - Reference [[behavior-contract]] for where descriptors live in compiled output

    6. **Pluggable Wire Format** — Interface definition:
       - `serialize(value, port_descriptor) → bytes`
       - `deserialize(bytes, port_descriptor) → value`
       - JSON implementation is the default
       - Future candidates: CBOR, MessagePack
       - No dynamic code generation — all implementations are pre-compiled

    7. **SDK Distribution** — Per-language packages:
       - Python: `polyglot-sdk` on PyPI
       - Rust: `polyglot-sdk` crate
       - Go: `polyglot.dev/sdk` module
       - JavaScript: `@polyglot/sdk` on npm
       - SDK version locked to Polyglot compiler version (same version number)

    8. **Design Decisions** — Key decisions with rationale:
       - Polyglot's type system is the bridge (no per-language type catalog)
       - All leaf values are strings
       - Shared memory = NoSQL variable store
       - No dynamic code generation — all SDK code is pre-compiled
       - **Why no FFI in the SDK:** FFI requires dynamic code generation to set up bindings between arbitrary language pairs, violating the "no dynamic code gen" principle. FFI is also not universally available across all supported languages. Trade-off: FFI is significantly faster than serialized string exchange. This is a deliberate choice — FFI optimization is deferred to -Run.Bridge (issue #321) as a separate, pairwise, language-specific concern.
       - Error propagation: SDK emits structured errors mapping to !Error tree
