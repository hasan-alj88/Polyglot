    Structure:
    1. YAML frontmatter: audience: [automation-builder, integrator, architect, designer], type: reference, updated: 2026-04-20
    2. <!-- @c:vision --> smart reference
    3. H1: Error Handling Philosophy
    4. Intro paragraph linking to [[vision]], framing error handling as Polyglot's most distinctive safety feature
    5. Five H2 sections, each grounded in specific Polyglot features:

    **Murphy's Law as Design Principle** — If something can go wrong, it will. Polyglot demands you plan for the worst. Reference the compiler's exhaustive coverage requirement. Connect to the trigger-driven async world where race conditions, network failures, and partial results are the norm, not the exception. Polyglot treats every failable operation as failable by default — the developer must handle the failure path, not hope it won't happen.

    **Handle It Now or Handle It at 3am** — Unhandled edge cases surface in production, guaranteed. The compiler is tiresome on purpose — cheaper to argue with the compiler during the day than debug a crashed pipeline at night. Reference how [!] blocks, fallback operators (!<, !>), and exhaustive conditionals force error handling at write time. The compiler catches missing handlers, missing collectors, missing branches — all the gaps that become 3am pages.

    **Failed Is a State, Not a Crash** — `Failed` is a variable lifecycle state (Declared → Default → Final → Failed → Released). Data still exists, the pipeline knows about it, error handlers can route/replace/escalate. Reference PGE02005 (Failed is terminal — no further pushes, but data accessible). Errors are data flowing through the tree, not exceptions blowing up the stack. This is fundamentally different from try/catch — there is no stack unwinding, no lost context, no "where did the error come from?" mystery. Reference [[concepts/variables/variable-lifecycle]] for the state machine.

    **No Happy Path Only Code** — Failable calls require error handling. Parallel jobs require collection. Conditionals require exhaustive branches (*? mandatory catch-all). Every expand must have a collect. Every race must account for all-Failed scenarios. Reference specific compile rules: PGE07008 (non-failable fallback), PGE03025 (all jobs released), PGE01040 (orphan parallel). Exhausting to write — that is the point.

    **The Compiler as Safety Net** — The compiler catches race conditions, resource conflicts, unhandled error paths, missing collectors, permission gaps. All compile-time errors, not runtime surprises. Reference how the compiler is a collaborator, not a gatekeeper — it tells you what you missed before production tells you. Connect to the "building permit" analogy from core-philosophy.md: compilation is a license to launch. The compiler rejects incomplete specifications, not because it is pedantic, but because the Polyglot Service needs instructions for every scenario.

    6. Related Philosophy section with [[wikilinks]] to all sibling pages

    Style: Follow the tone of accountability.md and cybersecurity.md — declarative, confident, no hedging. Use specific Polyglot terminology from the glossary. Reference technical docs with [[wikilinks]] where they ground the philosophy.

    Avoid:
    - Do not repeat full technical specifications — link to them
    - Do not use code examples (this is philosophy, not tutorial)
    - Do not introduce new concepts not already in the spec
