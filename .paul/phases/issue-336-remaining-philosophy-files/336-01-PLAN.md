    **All files share:**
    - Frontmatter: `audience: [automation-builder, integrator, architect, designer]`, `type: reference`, `updated: 2026-04-20`
    - `<!-- @c:vision -->` as first line after frontmatter
    - H1 title
    - Intro blockquote: `> This page [describes/expands]... See [[vision]] for the broader project context.`
    - H2 sections for each principle
    - Cross-references to related philosophy pages via [[wikilinks]]
    - NO code blocks — philosophy level, not spec level

    **data-trees.md:**
    - H1: Everything Is a Tree
    - Why trees: universal, composable, serializable, inspectable
    - Three-tier system: # (data tree), ## (schema/topology), ### (leaf properties)
    - Strings as universal values — RawString leaves, cross-language common tongue
    - How the tree model enables cross-language data exchange (topology known at compile time)
    - Link to [[data-is-trees]] for spec details

    **behavioral-contract.md:**
    - H1: The Behavioral Contract
    - "Building permit" metaphor expanded — every disaster scenario accounted for before launch
    - Single source of truth: compiler output = complete instructions for the Service
    - Properties: deterministic, inspectable before execution, versionable, auditable
    - What the contract captures: triggers, IO, error paths, concurrency resolution, permissions
    - Compiler→Service flow: compile = license to launch
    - Link to [[core-philosophy]] for the mind-shift context

    **developer-experience.md:**
    - H1: Developer Experience
    - The compile loop: write .pg → compile (get errors) → fix → compile (clean) → deploy
    - Compiler as collaborator, not gatekeeper — it catches what you missed
    - Why exhaustive checking helps: offloading runtime debugging to compile time
    - The "tiresome but correct" trade-off — verbose now, zero-surprise later
    - Contrast with traditional DX: debug-in-production vs fix-before-deploy

    **extensibility.md:**
    - H1: Extensibility
    - jm3lib as the standard library — battle-tested operations shipping with the language
    - Community packages: same compile-time validation as core
    - Permission ceiling: packages cannot exceed what the importing pipeline grants
    - **No resource hogging** — 100% utilization = system death; Polyglot enforces default breathing margins on RAM, CPU, GPU, disk, network; these margins account for other processes on the host. Exceeding resource limits requires explicit handling (same philosophy as error handling). Permissions are not just access control — they also govern compute resources.
    - Link to [[cybersecurity]] for the permission model

    **how-polyglot-differs.md:**
    - H1: How Polyglot Differs
    - Acknowledge what exists: Airflow (DAG scheduling), Temporal (workflow orchestration), Prefect (data pipelines), gRPC (cross-language RPC)
    - What they share: they are add-ons bolted onto general-purpose languages — afterthoughts, not foundations
    - What Polyglot does differently:
      1. Cross-language integration as language feature, not library
      2. Compile-time safety for concurrent workflows
      3. Permissions and security as first-class constructs
      4. **Resource handling as a feature** — breathing margins, resource limits as permissions, host-aware load balancing via queue conditions. Automation users can set queue conditions to select least-busy host or any other criteria, bringing automation to load balancing itself via Polyglot code. No other automation tool treats compute resources as manageable, permission-controlled objects.
    - Tone: respectful positioning — these tools are good at what they do, Polyglot addresses what they don't

    **Avoid:**
    - Duplicating spec-level technical details (EBNF, compile rules, metadata paths)
    - Code examples — philosophy pages use prose and concepts
    - Making claims about performance or speed
    - Disparaging competing tools
