         - Presence of `.baseCode` makes the definition a **base pipeline**: no execution body allowed (`[T]`/`[Q]`/`[W]`/`[r]`/`[p]`/`[b]`/`[s]` forbidden). Only `[=]` IO and `[!]` error declarations are permitted alongside `[%]`.

         - The `#BaseCode` variant must exist and use the configured base language (PGE01028).

    2. **Update `docs/user/concepts/pipelines/INDEX.md`**:
       - Add "## Base vs Derived" section AFTER "## Marker Declarations" and BEFORE "## Sub-Pages".
       - Content:
         - **Base pipelines**: defined in aj3lib `.pg` files with full interface (`[=]` IO, `[!]` errors, `[%]` metadata) but no execution body. A `[%] .baseCode << #BaseCode.Rust.*` line links to the native compiler implementation.
         - **Derived pipelines**: have full Polyglot execution bodies (`[T]`, `[Q]`, `[W]`, `[r]`/`[p]`/`[b]`). No `.baseCode` field.
         - Mutual exclusion: `.baseCode` + body = compile error (PGE01028). Body without `.baseCode` = derived. No body without `.baseCode` = error (except `{T}`/`{Q}` which are IO-only by design).
         - Config: `base: Rust` selects the active base language.

<task type="auto">
  <name>Task 3: Create PGE01028 base/derived mutual exclusion compile rule</name>
  <files>docs/technical/compile-rules/PGE/PGE01028-base-derived-mutual-exclusion.md</files>
  <action>
    Create PGE01028 following the format of existing PGE rules (see PGE01005):
    - Frontmatter: rule "1.28", code PGE01028, name "Base/Derived Pipeline Mutual Exclusion", severity error
    - Statement: Base and derived pipeline constraints are mutually exclusive. A pipeline with `.baseCode` cannot have an execution body; a pipeline with a body cannot have `.baseCode`.
    - Rationale: Base pipelines delegate to native code — a Polyglot body would conflict. Derived pipelines are pure Polyglot — `.baseCode` would create ambiguity about which implementation runs.
    - Detection: The compiler checks for mutual exclusion between `.baseCode` metadata and execution body elements.
    - Define 4 sub-conditions:
      a. `.baseCode` + execution body = error (applies to `{=}`, `{T}`, `{W}`, `{Q}`)
      b. Execution body + `.baseCode` = error (same check, different framing)
      c. No body + no `.baseCode` = error for `{=}` (bodyless pipeline must be base). Exception: `{T}` and `{Q}` are IO-only by design — no body needed.
      d. `.baseCode` references non-existent `#BaseCode` variant = error
      e. `.baseCode` uses wrong base language vs config = error
    - VALID examples: base `{=}[exe] =File.Text.Read` with .baseCode and no body, derived `{=} =ProcessData` with body and no .baseCode, `{T} =T.Call` base trigger
    - INVALID examples from brainstorming Rule F: base+body, derived+.baseCode, wrapper variants, invalid variant, wrong language
  </action>
