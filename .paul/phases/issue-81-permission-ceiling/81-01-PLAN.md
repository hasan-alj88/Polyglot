<task type="auto">
  <name>Task 1: Add Permissions section to packages.md</name>
  <files>docs/user/syntax/packages.md</files>
  <action>
    Read packages.md first.

    Add a new "## Permissions" section after "## Multi-File Packages" and before "## Compile Rules Reference".

    Content:
    1. Opening paragraph: `{@}` block can declare `[_]` permission ceiling — maximum IO permissions any definition in the package can request. Cross-ref [[permissions]].

    2. Example showing ceiling in {@} block (from issue):
       ```polyglot
       {@} @Local:999.LogAnalyzer:v1.0.0
          [@] @http << @Community:devops.HttpClient:v2.1.0
          [_] _File.read"/var/log/*"
          [_] _File.write"/tmp/reports/*"
          [_] _Web.request
             [_] <url;string << "https://alerts.internal/*"
             [_] <method;string << "POST"
          [_] _System.env"LOG_LEVEL"
       ```

    3. Rules subsection:
       - **Ceiling, not grant** — `[_]` in `{@}` sets the maximum. Each `{=}` or `{M}` must explicitly request permissions it needs. Nothing is inherited.
       - **No ceiling = no IO** — if `{@}` has no `[_]` lines, the entire package is pure computation. Any IO call in any pipeline is a compile error (PGE-915).
       - **Pipeline subset** — every `[_]` in a pipeline must fall within the package ceiling. A pipeline requesting `_File.read"/etc/shadow"` when the ceiling only allows `_File.read"/var/log/*"` is PGE-915.
       - **Import ceiling** — the compiler checks each imported package's `{@}` ceiling against the importer's ceiling. If the imported package declares permissions outside the importer's ceiling, it's PGE-916.
       - **[_] placement** — `[_]` lines go after `[@]` imports and before any `{=}`/`{#}` definitions in the `{@}` block.

    4. Add PGE-915 and PGE-916 to the Compile Rules Reference table:
       - PGE-915 | Pipeline Exceeds Package Permission Ceiling | [[#Permissions]]
       - PGE-916 | Imported Package Exceeds Importer Permission Ceiling | [[#Permissions]]

    Do NOT modify any existing sections — only add the new Permissions section and table rows.
  </action>
  <verify>Grep for "## Permissions" in packages.md; confirm PGE-915 and PGE-916 in compile rules table; confirm [[permissions]] wikilink</verify>

<task type="auto">
  <name>Task 2: Add pipeline-level [_] to pipelines.md</name>
  <files>docs/user/concepts/pipelines.md</files>
  <action>
    Read pipelines.md first to understand its structure.

    Add a "### Permissions" subsection in the appropriate location — near the pipeline structure/anatomy section, showing where [_] sits in a pipeline definition.

    Content:
    1. Pipelines can declare `[_]` permission lines after the `{=}` header and before `[t]`, `[Q]`, `[W]`, and IO. Same applies to all `{x}` definitions (`{M}`, etc.).

    2. Example:
       ```polyglot
       {=} =AnalyzeLogs
          [_] _File.read"/var/log/*.log"
          [t] =T.Manual
          [Q] =Q.Default
          [W] =W.Polyglot
          [=] <logPath;path
          [=] >summary;string
          [r] $content << =File.Text.Read >> "{$logPath}"
          [r] >summary << ...
       ```

    3. Rules:
       - Pipeline `[_]` must be a subset of the package `{@}` ceiling (PGE-915)
       - No `[_]` in pipeline = pure computation, even if the package has a ceiling
       - See [[permissions]] for the full permission system and [[packages#Permissions]] for ceiling rules

    Do NOT modify any existing pipeline content — only add the new subsection.
  </action>
