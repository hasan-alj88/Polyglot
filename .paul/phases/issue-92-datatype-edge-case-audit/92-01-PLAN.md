<objective>
## Goal
Audit every `{#}` datatype definition in the docs for edge cases, verify `%##` property completeness, test `<~` inheritance for parameterized types, and split the monolithic stdlib types.md into per-category files in the types/ folder.

## Purpose
Issue #88 introduced `%##` schema properties and the three-tier prefix system. The stdlib types.md is stale — it uses inconsistent notation (mixed `%Property` and `%##Property`), has old-style enum definitions (no `{#}` block), and still includes #Dataframe which was dropped. A systematic audit + restructure brings stdlib type docs up to current spec quality.

## Output
- New edge case section in EDGE-CASES.md for datatype definitions
- Stdlib types/ folder restructured into per-category files
- All type definitions using current notation (consistent `%##` prefix, `{#}` blocks)
</objective>

    - Test regex boundaries: empty string, whitespace, leading zeros, sign edge cases
    - #Dimension: DECISION — allow 0D for scalars. Correct regex is "^[0-9]+$" (no trailing D — the D is syntax sugar in usage like :2D, not part of the value). Both stdlib and syntax files need fixing.

    - #path: two OS subfields — what if only one assigned?
    - #Queue: struct containing enum-typed fields — mixed ###Value and ###Enum at different tree levels

    **For collection types** (#Map, #Array, #Serial, #Dataframe):
    - #Array <~ #Map parameterized inheritance: does #Array get #Map's %## properties? How does ##Contiguous override ##Sparse?
    - 0D array: valid? What is Dim=0?
    - Empty collections: 0-element #Array, 0-entry #Map
    - #Serial: no ## constraints — is PGW-906 relevant for stdlib?
    - #Dataframe: still in stdlib types.md but syntax/types.md doesn't list it. DECISION — TBD. Document status as edge case; do not remove or keep yet.
    - #Map<#Int<#String: #Int does NOT inherit #KeyString — should this PGE-924?

    - List which %## properties each type needs but doesn't explicitly have
    - Note that #Array and #Serial in stdlib types.md use %Key.Type, %Key.Gap, %Ordered — these should be %##Children.Type, %##Children.Gap, %##Children.Ordered

    Format: use existing EDGE-CASES style (EBNF ref, What it tests, Cross-refs, Example).
