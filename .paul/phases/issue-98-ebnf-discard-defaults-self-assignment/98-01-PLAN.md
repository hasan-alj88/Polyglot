<objective>
## Goal
Add compile errors and EBNF restrictions for three meaningless assignment patterns: default assignment to discard (`$* <~`), self-assignment via output params (`>x << >x`), and variable self-assignment (`$x << $x`).

## Purpose
These patterns are syntactically valid but semantically meaningless. Catching them at compile time prevents no-op code from reaching runtime and helps developers spot wiring mistakes.

## Output
- 2 new PGE compile rule files (PGE02010, PGE08011)
- Tightened EBNF in 08-expressions.md
- 3 new edge cases in 08-expressions.md (EC-8.4, EC-8.5, EC-8.6)
- Updated COMPILE-RULES.md index and edge-cases INDEX.md
</objective>

  <action>
    **PGE02010 — Discard Default Assignment** (category 02 — Lifecycle & State):
    - Rule 2.10, code PGE02010, severity error
    - Statement: `$*` (inline discard) only accepts final assignment operators (`<<`, `>>`). Default operators (`<~`, `~>`) on `$*` are compile errors because the value is immediately released — a default implies the value may be overridden later, which is meaningless for a discard.
    - VALID: `$*#string << "discarded"`, `>result >> $*`
    - INVALID: `$*#string <~ "never used"`, `>result ~> $*`
    - See also: PGE02003 (Final Is Push-Once), PGW02002 (Unused Variable)

    **PGE08011 — Self-Assignment** (category 08 — Chains & IO Wiring):
    - Rule 8.11, code PGE08011, severity error
    - Statement: An assignment where the target and source are the same identifier is a no-op and a compile error. This applies to: (a) output params — `>x << >x` where same param name in same operation scope, (b) variables — `$x << $x` where same variable name. Cross-port wiring (`>out1 << >out2`) is valid when the names differ.
    - VALID: `>out1 << >out2` (different ports), `$a << $b` (different vars)
    - INVALID: `>result << >result` (same output param), `$name << $name` (same variable)
    - See also: PGE02010 (Discard Default Assignment)

    - Add `| PGE08011 | Self-Assignment |` after PGE08010 in category 08
