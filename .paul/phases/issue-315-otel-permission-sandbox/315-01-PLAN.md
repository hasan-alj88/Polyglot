<objective>
## Goal
Create the OpenTelemetry specification for permission and sandbox events — the 8 log event types, structured attributes, span hierarchy, and runtime compliance report integration.

## Purpose
Completes the observability layer for the permission/sandbox system (#313/#314). Without structured logging, sandbox violations, resource limit breaches, and opaque code activations are invisible to operators. This spec defines *what* is logged and *when*; the foundation issue (#318) covers *how* (SDK, exporters, config).

    3. **Event Table**: All 8 events from #315 issue body:
       - permission.sandbox.violation (ERROR)
       - permission.sandbox.setup (INFO)
       - permission.sandbox.setup_failed (CRITICAL)
       - permission.resource.exceeded (WARN/ERROR — WARN for throttle, ERROR for kill)
       - permission.resource.kill (ERROR)
       - permission.ast.suppressed (WARN)
       - permission.opaque.activated (WARN)
       - permission.compliance.generated (INFO)

    4. **Per-event sections**: For each event:
       - Trigger condition (exactly when this event fires)
       - Severity and rationale
       - Required attributes (from the polyglot.* namespace)
       - Optional attributes (event-specific extras)
       - Example OTel log record (JSON-like structured format)
       - Which span this event attaches to

    5. **Attribute Registry**: Table of all polyglot.* attributes used by these events:
       - polyglot.job.uid (string)
       - polyglot.pipeline.name (string)
       - polyglot.package.name (string)
       - polyglot.permission.category (string — {_} category name)
       - polyglot.sandbox.layer (string — landlock/seccomp/namespace/cgroup)
       - polyglot.sandbox.syscall (string — blocked syscall name)
       - polyglot.sandbox.resource (string — attempted resource path/host:port/bytes)
       - polyglot.sandbox.action (string — #Kill/#Throttle/#Retry/blocked)
       - polyglot.sandbox.opaque (bool — _Unsafe.SandboxOnly active)

    6. **Span Hierarchy**: Sandbox-specific span tree showing:
       ```
       Job Span (job:{UID})
         ├── Sandbox Setup Span (permission.sandbox.setup)
         │     └── permission.sandbox.setup_failed (if any)
         ├── Foreign Code Execution Span
         │     ├── permission.sandbox.violation (if any)
         │     └── permission.resource.exceeded (if any)
         ├── permission.resource.kill (if any)
         └── Job Completion Span
             └── permission.compliance.generated
       ```
       Note: This nests inside the Pipeline Span from #318.

    7. **Runtime Compliance Appendix Format**: Define structured format for runtime violation data appended to the compile-time compliance report:
       - Violation count, warning count
       - Per-violation: timestamp, event type, job UID, attributes
       - Stored alongside the Behavior Contract in NoSQL DB

    8. **Exporter Independence**: Note that this spec defines events and attributes only. Where logs go (OTLP endpoint, stdout, JSON files) is configured by the user per #318. The Runner emits OTel data; the exporter routes it.

    9. **Open Questions** section (from #315):
       - Fallback logging: stderr JSON as backup when OTel exporter unavailable?
       - Sampling: sandbox setup spans sampled differently from normal execution?
       - NATS trace context: should NATS messages carry traceparent for cross-service correlation?

    Follow docs/audit/ conventions: proper heading levels, code blocks with language tags, no bare fences.
