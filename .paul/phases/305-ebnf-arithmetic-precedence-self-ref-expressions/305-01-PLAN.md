<objective>
## Goal
Remove `arithmetic_expr` and `arithmetic_op` productions from EBNF (already banned by PGE04010), break the `inline_data` → `value_expr` cycle by introducing a non-recursive `inline_value` production, and add PGE08013 compile rule for nested inline data.

## Purpose
X.35: Raw arithmetic is already a compile error (PGE04010) — the EBNF shouldn't define productions for always-invalid syntax. EC-6.4 contradicts PGE04010 by showing arithmetic as valid.

X.36: The `inline_data → value_expr → inline_data` cycle allows unbounded nesting (`{{1,2},{3,4}}`). User decision: no nested inline data. Break the cycle with a flat `inline_value` production.

## Output
- EBNF §6.4 and §8.3 removed
- `value_expr` no longer includes `arithmetic_expr`
- `inline_data` uses `inline_value` (non-recursive) instead of `value_expr`
- EC-6.4 rewritten as INVALID examples (aligned with PGE04010)
- New edge cases EC-8.7/EC-8.8 for X.35/X.36
- PGE08013 compile rule for nested inline data
</objective>

    Create PGE08013-nested-inline-data.md:
    - Frontmatter: audience: developer, rule: "8.13", code: PGE08013, name: Nested Inline Data, severity: error
    - Statement: Inline data literals (`{...}`) cannot contain other inline data literals. The `inline_data` production uses `inline_value` (not `value_expr`), which excludes nested `{}`. Nested collection construction requires pipeline calls (e.g., building inner arrays first, then composing).
    - Rationale: Nested inline data creates parse ambiguity (is `{{1}}` an empty record with `{1}` inside, or a nested singleton array?). Flat literals are unambiguous; complex structures use pipelines.
    - Detection: Parser encounters `{` inside an `inline_data` context. PGE08013 fires.
    - VALID examples: flat inline data like `{1, 2, 3}`, `{"a", "b"}`
    - INVALID examples: `{{1,2},{3,4}}`, `{{{1}}}`, `{1, {2, 3}, 4}`
    - See also links to inline_data EBNF and EC-8.8

    - Format: `| PGE08013 | Nested Inline Data | Nested `{}` inside inline data | error |`
