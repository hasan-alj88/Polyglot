<objective>
## Goal
Create the {$} Constructor block specification as a new first-class definition type in Polyglot, and update all user-facing documentation to reflect the new block type, its elements, and the scoping of inline strings to infrastructure lines only.

## Purpose
Constructors eliminate the error-handling ceremony for compile-time-provable value construction. This separates two fundamentally different operations that previously shared the same inline pipeline syntax: pure data transformation (constructor) vs async operations with side effects (pipeline call).

## Output
- New specification: `docs/user/syntax/constructors.md`
- Updated: `blocks.md` ({$} definition + ($)/[$] elements)
- Updated: `inline-calls.md` (scoped to [T]/[Q]/[W] infrastructure only)
- Updated: `glossary.md` (Constructor term)
- Updated: `SPEC-INDEX.md` (constructor link)
</objective>

## Design Decisions from Discussion
- `($)` for IO lines (regex input parameters), `[$]` for action lines (type binding) — three-bracket consistency
- `%InlineString` retired from `{-}` pipelines; only `{T}` triggers keep inline string config
- Scoped extension model: any package can define constructors for external types, scoped by `[@]` imports
- `.re` is mandatory on all capture parameters (for validation AND regex matching)
- Compiler grants infallibility trust to native pipeline overloads

<task type="auto">
  <name>Task 1: Create {$} constructor specification</name>
  <files>docs/user/syntax/constructors.md</files>
  <action>
    Create new spec file with audience: automation-builder frontmatter. Sections:

    1. **Introduction** — what constructors are, why they exist (compile-time guaranteed Final values, no error surface), the split from inline pipelines

    2. **{$} Block Syntax** — definition header `{$} $Name"pattern"`, two overload forms:
       - String-parsing overload: `($) <param.re << "regex"` for captures, `[$] #TargetType` for type binding, `[.] .field << <param` for field mapping
       - Native pipeline overload (aj3lib only): `[-]` infallible pipeline call, then `[$]` + `[.]`

    3. **Constructor Contract** — guaranteed Final, no Failed state, compiler-proven exhaustive outcomes, guaranteed existence not guaranteed value

    4. **Overload Resolution** — regex compilation from pattern + `.re` substitution, exactly-one-match rule, ambiguous overlap = definition compile error, keyword overloads compile to exact-match regex

    5. **Structural Integrity Check** — at definition compile time, verify no slot's `.re` can match pattern's literal separators (SQL injection prevention)

    6. **Interpolation Rule** — `{$var}` allowed only if `$var` is constructor-sourced, runtime/IO-sourced = compile error, pure substitution only

    7. **Three-Context Rule** — table: Infrastructure ([T]/[Q]/[W]) uses inline pipeline config, Body known values use constructors, Body dynamic values use pipeline calls

    8. **Cross-Package Constructors** — scoped extension model: any package can define {$} for external types, visibility scoped by [@] imports, no cross-package ambiguity

    9. **Usage Examples** — constructor invocation in pipeline body with `[-] $var << $Constructor"literal"`, interpolation chaining, side-by-side with pipeline call showing error handling contrast

    10. **Metadata Tree** — `%definition.$.*` for definitions (overloads by index), `%{instance}.$.*` for instances

    Use @-imports to blocks.md, inline-calls.md, error-philosophy.md. Use [[wikilinks]] for Obsidian navigation.
    Do NOT include compile error codes (deferred to #343).
    Do NOT include aj3lib constructor catalog (deferred to #342).
  </action>
