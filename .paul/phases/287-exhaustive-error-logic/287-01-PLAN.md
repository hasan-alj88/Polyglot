<objective>
## Goal
Specify the exhaustiveness algorithm for PGE07007, add grouped `(-) $label` fallback syntax, resolve PGE02005/PGE07007 overlap, and establish the strict rule: no variable may compile if it can reach Failed state.

## Purpose
Error exhaustiveness is Polyglot's core safety guarantee — every error path must be explicitly handled. The current docs state the rule but don't specify the algorithm. This plan makes PGE07007 implementation-ready for the compiler.

## Output
- PGE07007 expanded with exhaustiveness algorithm, chain rules, multi-output rules
- PGE02005 narrowed to variable lifecycle (delegates exhaustiveness to PGE07007)
- errors.md updated with grouped `(-) $label` fallback syntax + examples
- EBNF updated with grouped fallback grammar
- Cross-references corrected (PGE02005 → PGE07007 where appropriate)
</objective>

## AC-1: Exhaustiveness Algorithm Specified
```gherkin
Given a failable pipeline call with N declared errors
When the compiler checks for exhaustive error handling
Then PGE07007 fires if any declared error is not covered by the union of:
  - [!] specific handler blocks
  - [!] !* wildcard handler
  - (>) !ErrorName> specific fallbacks (on all outputs)
  - (>) !> generic fallbacks (on all outputs)
  - ($) grouped fallbacks under (-) $label
And the algorithm is documented step-by-step in PGE07007
```

## AC-2: Strict No-Failed-State Rule
```gherkin
Given any variable that receives its value from a failable call
When the compiler performs static analysis
Then compilation fails if there exists any code path where the variable can reach Failed state without explicit handling
And PGE07007 diagnostic message includes the unaddressed error name, pipeline name, and line number
```

## AC-3: Chain Error Exhaustiveness
```gherkin
Given a chain call [-] -A->-B->-C where each step declares errors
When the compiler checks exhaustiveness
Then [!] !* (global wildcard) covers errors from ALL chain steps
And .N!* (step-scoped wildcard) covers only step N's errors
And the algorithm is documented in PGE07007 with chain-specific examples
```

## AC-4: Grouped Fallback Syntax
```gherkin
Given a pipeline call with multiple outputs
When the author uses (-) $label with ($) fallback lines
Then each ($) >outputName !> value provides a fallback for that output
And ($) >outputName !ErrorName> value provides error-specific fallback
And [!] blocks can appear under (-) $label
And the grouped form is semantically equivalent to scattered (>) fallbacks
And EBNF grammar in §10 includes the grouped fallback production
```

## AC-5: PGE02005/PGE07007 Overlap Resolved
```gherkin
Given PGE02005 (Failed Must Resolve) and PGE07007 (Error Handling Must Be Exhaustive)
When a developer reads either rule
Then PGE07007 owns the exhaustiveness algorithm and all error coverage logic
And PGE02005 focuses on variable lifecycle: Failed is terminal, recovery produces Final
And cross-references in errors.md and error-handling.md point to PGE07007 for exhaustiveness (not PGE02005)
```

## AC-6: Multi-Output Coverage
```gherkin
Given a failable pipeline call with outputs >out1, >out2, >out3
When only >out1 and >out2 have error fallbacks
Then PGE07007 fires for >out3 — all outputs must be addressed
And the diagnostic names the uncovered output port
```

  <action>
    Rewrite PGE07007 to include:

    1. **Exhaustiveness Algorithm** (new section after Detection):
       - Step 1: Collect declared errors E = {!E1, !E2, ...} from called pipeline's (-) ! declarations
       - Step 2: Collect handled errors H from [!] specific blocks
       - Step 3: If [!] !* present, H = E (wildcard covers all)
       - Step 4: Collect fallback-covered errors F from (>) lines on each output
       - Step 5: Collect grouped fallback errors G from (-) $label / ($) lines
       - Step 6: For each output port, compute coverage = H ∪ F ∪ G
       - Step 7: If coverage ⊊ E for any output, emit PGE07007 naming the uncovered errors

    2. **Strict rule statement**: "No variable may compile if there is a non-zero probability it can reach Failed state without explicit handling."

    3. **Chain exhaustiveness** (new subsection):
       - Global [!] !* covers all steps' errors
       - Step-scoped .N!* covers only step N
       - Each step's errors independently tracked: E = E_0 ∪ E_1 ∪ ... ∪ E_n
       - Step-scoped handlers address only their step's subset
       - Example with mixed step-scoped + global wildcard

    4. **Multi-output rule** (new subsection):
       - Every output port from a failable call must have coverage
       - [!] blocks that push to all outputs satisfy all ports
       - (>) fallbacks only cover the output they're declared on
       - Diagnostic names the uncovered output port

    5. **Updated diagnostic format**:
       - Single output: "Unaddressed error `!ErrorName` from `-PipelineName` at line N — add [!] handler, [!] !* wildcard, or (>) !> fallback"
       - Multi-output: "Output `>portName` has unaddressed error `!ErrorName` from `-PipelineName` at line N"
       - Chain: "Unaddressed error `.N!ErrorName` from chain step N (`-StepName`) at line N"

    6. Keep all existing VALID/INVALID examples. Add new examples for:
       - Grouped (-) $label fallback (VALID)
       - Multi-output partial coverage (INVALID)
       - Chain with mixed step-scoped + global wildcard (VALID)
       - [b] fire-and-forget exemption stays as-is

    Model the algorithm section structure on PGE06001's "Exhaustiveness by Type" table.
