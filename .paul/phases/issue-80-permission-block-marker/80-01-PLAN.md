<objective>
## Goal
Add `_` as a permission identifier prefix and `[_]` as the permission block marker to the Polyglot language specification. Create a dedicated permissions concept document covering inline/IO forms, permission categories, hierarchical scoping, and compile-time enforcement.

## Purpose
Foundation for Polyglot's implicit-deny permission system (Cisco ACL model). Without explicit `[_]` declarations, pipelines are pure computation with zero IO — security by default.

## Output
- `docs/user/syntax/identifiers.md` — `_` prefix added to prefix table
- `docs/user/syntax/blocks.md` — `[_]` marker added with Permissions category
- `docs/user/concepts/permissions.md` — new spec file for the permission system
</objective>

<task type="auto">
  <name>Task 3: Create permissions.md concept spec</name>
  <files>docs/user/concepts/permissions.md</files>
  <action>
    Create new spec file with standard frontmatter (audience: user, type: specification, status: complete).

    Sections to include:

    1. **Overview** — Implicit-deny model. No `[_]` = pure computation, zero IO. Cisco ACL analogy.

    2. **Permission Identifiers** — `_` prefix with `.` fixed-field navigation. Categories use dot-namespaced names: `_File.read`, `_Web.request`. Cross-ref [[identifiers]].

    3. **Inline Form** — Single-argument permissions: `[_] _File.read"/var/log/*"`. Like inline pipelines (`=Path"/tmp"`).

    4. **IO Form** — Multi-parameter permissions with input-only IO:
       ```polyglot
       [_] _Web.request
          [_] <url;string << "https://api.example.com/*"
          [_] <method;string << "GET"
       ```
       IO is input-only — `[_] <param` only, no `>` output.

    5. **Permission Categories** — Table of all 8 categories with subfields:
       _File (.read, .write, .execute, .delete),
       _Web (.request IO, .socket IO),
       _Database (.connect IO, .read, .write),
       _System (.env, .process IO, .signal),
       _Crypto (.key, .sign, .encrypt),
       _IPC (.send IO, .receive IO, .subscribe),
       _Device (.camera, .microphone, .location, .bluetooth),
       _Memory (.allocate, .shared)

    6. **Hierarchical Scoping** — `[_]` in `{@}` sets package ceiling (allows but does not grant). `[_]` in `{=}`, `{M}`, etc. must explicitly request permissions (can only narrow, never widen). No inheritance — must restate or alias.

       Example showing ceiling vs request vs denial.

    7. **Compile-Time Enforcement** — All permissions are statically verifiable. Compiler checks:
       - Pipeline uses `=File.Text.Read` → must have `[_] _File.read` permission
       - Pipeline `[_]` requests must fall within `{@}` ceiling
       - No `[_]` in pipeline = pure computation (any IO call is compile error)

    8. **Foreign Code** — `[c]` blocks under permissions: compiler warning (not error) that foreign code cannot be statically verified. Programmer takes responsibility. Foreign runtime handles its own enforcement.

    Cross-reference [[blocks]] for `[_]` marker, [[identifiers]] for `_` prefix.
    Use [[wikilinks]] for Obsidian compatibility throughout.
  </action>
