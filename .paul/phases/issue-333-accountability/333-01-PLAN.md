    **Frontmatter:**
    - audience: [automation-builder, integrator, architect, designer]

    **Header:** `<!-- @c:vision -->` import, then H1 "Accountability" with intro quote referencing [[vision]].

    **Sections (in order):**

    1. **The Accountability Chain** — Five-stage chain: Author -> Inspector (human) -> Compiler -> Permission Grant -> Execution. Each stage gets 1-2 sentences explaining its role. The key insight: every stage is a gate, and no gate can be skipped. The compiler is powerful but it is not a substitute for human judgement.

    2. **Human Inspection Is Non-Negotiable** — Correctness is not accountability. A compiler can verify that code is syntactically valid and type-safe, but it cannot verify intent. A human inspector takes personal responsibility for what the code does. This is not about trust in tools — it is about requiring a named person who stands behind the decision to run this code.

    3. **Why No Runtime-Generated Code?** — Three subsections:
       - **Security** — Unverified intent, no permission checks, obvious breach vector. Dynamic code bypasses the entire Accountability Chain.
       - **Accountability and Auditability** — All code must be reviewable by a human auditor who takes responsibility. Runtime-generated code cannot be reviewed before it runs.
       - **Black Box Accountability** — `author` and `auditor` metadata fields become **mandatory** (not optional) for compiled binaries. A named person is accountable for sloppy audits. This closes the "nobody's fault" loophole.

    4. **AI Policy** — Three rules, no exceptions:
       1. AI-generated code is untrusted by default — human must inspect at compile time
       2. Auditing and inspection always done by a human who shoulders accountability
       3. Even if AI verification becomes reliable, a human must give the green light — design principle, not technology limitation

    5. **Compile-Time File Binding** — Content-bound permissions. Any pipeline input referencing a source file has its permission grant bound to the file's content hash at compilation. If the file changes, permissions are revoked and the pipeline refuses to execute until recompiled. Cross-reference [[concepts/permissions/enforcement#Compile-Time File Binding]] and [[technical/spec/behavior-contract]].

    6. **Related Philosophy** — Wikilinks to [[philosophy/core-philosophy]], [[philosophy/language-design]], [[philosophy/cybersecurity]] (planned -- #334).

    **Style guidance:**

    - Use bold for key terms on first use
    - Keep each section concise — this is a philosophy page, not a technical spec
    - Do NOT duplicate technical details from ast-invisible-registry or compile rules — cross-reference them
    - Do NOT add Mermaid diagrams (keep text-focused like other philosophy pages)
