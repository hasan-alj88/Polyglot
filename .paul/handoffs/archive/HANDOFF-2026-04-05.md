## Design Decisions (from brainstorm — CRITICAL for APPLY)

1. `{_}` is a first-class block type for named, reusable permission objects
2. `_`/`__`/`___` tiers mirror `#`/`##`/`###` (object / descriptor / constraint)
3. `.intent` field: `#Ceiling` allows globs, `#Grant` requires specific narrow values
4. `[_]` in `{@}` and `{=}` always references a `{_}` object by name — no inline declarations
5. Narrowing allowed (pipeline restricts policy subset), expanding NOT allowed
6. `_` objects must be fully filled — no empty leaves (compile error)
7. No instances — permissions are compile-time, apply across all pipeline instances
8. Per-category capability enums: `#FileCapability`, `#WebCapability`, etc.
9. `{@}` first is compiler-enforced; `{_}` → `{#}` → `{=}` is recommended style
10. `__Permission` is the schema; `___Unix`, `___Sandboxed` are constraint descriptors

## Example Syntax (approved by user)

```
{_} _DataCeiling
   [.] .intent << #Ceiling
   [.] .File.Read "data/*.csv"
   [.] .Database.Read "*.postgres"

{_} _ReportReader
   [.] .intent << #Grant
   [.] .File.Read "data/reports/q1.csv"
   [.] .File.Read "data/reports/q2.csv"

{@} MyPackage
   [_] _DataCeiling

{=} =ProcessQ1
   [_] _ReportReader
```
