# Branch Specifications

<!-- @source:metadata-tree/INDEX -->

This file specifies the pipeline, wrapper, queue, trigger, and permission branches of the metadata tree.

## Pipeline Branch

`%=` stores pipeline instances (`{=}`). Each concurrent run of the same pipeline definition creates a new instance with sequential numbering (`:0`, `:1`, `:2`). Within each instance, jobs are tracked by UID ‚Äî not sequential numbers.

### Structure

```
%=:MyPipeline:0
‚îú‚îÄ‚îÄ .<                           <- input ports
‚îÇ   ‚îî‚îÄ‚îÄ .data#Serial
‚îú‚îÄ‚îÄ .>                           <- output ports
‚îÇ   ‚îî‚îÄ‚îÄ .result#Serial
‚îú‚îÄ‚îÄ .jobs                        <- job instances (UID-keyed)
‚îÇ   ‚îú‚îÄ‚îÄ :abc123#Job              <- individual job
‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ .PID
‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ .status
‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ .hierarchy           <- e.g., "MyPipeline/abc123"
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ ...
‚îÇ   ‚îî‚îÄ‚îÄ :def456#Job
‚îÇ       ‚îú‚îÄ‚îÄ .parent              <- "abc123" (parent job)
‚îÇ       ‚îî‚îÄ‚îÄ ...
‚îî‚îÄ‚îÄ .status                      <- pipeline instance status
```

### Key Properties

- **Sequential instances** ‚Äî each concurrent run creates `%=:Name:N` with sequential numbering. Instance `:0` is the first run, `:1` the second, etc.
- **UID-keyed jobs** ‚Äî within each instance, `.jobs` contains jobs identified by UID (generated by the Trigger Monitor), not sequential numbers. This allows the queue system to track jobs by globally unique ID.
- **IO boundaries define jobs** ‚Äî the compiler determines job start/end via pipeline IO state: inputs must be Final to start, outputs Final or Failed to complete. All actions and reactions are fully deterministic once compiled.
- **Job hierarchy** ‚Äî jobs form a parent/child tree within the instance. `.hierarchy` stores the dot-separated path (e.g., `ProcessData/job1/job3`). `.parent` references the parent job UID.
- **`live` fields** ‚Äî pipeline instances report runtime state: `status`, IO port values. See [[metadata|user/concepts/metadata]].

## Wrapper Branch

`%W` stores wrapper definitions (`{W}`). Wrappers provide setup/cleanup scope around pipeline execution bodies. Each `[W]` invocation in a pipeline creates a new wrapper instance.

### Structure

```
%W:DB.Connection:0
‚îú‚îÄ‚îÄ .[{]                     <- inputs from calling pipeline
‚îÇ   ‚îî‚îÄ‚îÄ .connectionString#string
‚îú‚îÄ‚îÄ .[}]                     <- outputs exposed to calling pipeline
‚îÇ   ‚îî‚îÄ‚îÄ .dbConn
‚îú‚îÄ‚îÄ .setup                   <- [\] setup phase
‚îî‚îÄ‚îÄ .cleanup                 <- [/] cleanup phase
```

### Key Properties

- **Flexible instances** ‚Äî each `[W]` invocation creates `%W:Name:N` with sequential numbering, like pipelines.
- **IO via `[{]`/`[}]`** ‚Äî wrapper inputs (`[{]`) and outputs (`[}]`) are fixed typed data sections, analogous to `.<`/`.>` in pipelines.
- **Composite wrappers** ‚Äî a `{W}` definition can contain `[W]` references to other wrappers inside `[\]` or `[/]`, creating nested wrapper instances.
- **`live` fields** ‚Äî wrapper instances report runtime state: `status`, `errors`, `setupDuration`. See [[metadata|user/concepts/metadata]].

## Queue Branch

`%Q` stores queue definitions (`{Q}`). Queues manage pipeline dispatch ordering and concurrency. Each queue dispatches pipelines and tracks active/pending counts.

### Structure

```
%Q:GPUQueue:0
‚îú‚îÄ‚îÄ .strategy#QueueStrategy        <- FIFO, LIFO, Priority
‚îú‚îÄ‚îÄ .host#String                   <- target host (1 queue = 1 host)
‚îú‚îÄ‚îÄ .maxInstances#UnsignedInt      <- max parallel instances per pipeline
‚îú‚îÄ‚îÄ .maxConcurrent#UnsignedInt     <- max other pipelines alongside
‚îú‚îÄ‚îÄ .resourceTags#Array:ResourceTag <- resource constraint tags
‚îú‚îÄ‚îÄ .killPropagation#KillPropagation <- Cascade or Downgrade
‚îú‚îÄ‚îÄ .maxWaitTime#String            <- max time before escalation
‚îú‚îÄ‚îÄ .description#String            <- human-readable description
‚îî‚îÄ‚îÄ .controls                      <- active queue controls
    ‚îú‚îÄ‚îÄ .pause
    ‚îú‚îÄ‚îÄ .resume
    ‚îî‚îÄ‚îÄ .kill
```

### Key Properties

- **Flexible instances** ‚Äî each queue use creates `%Q:Name:N` with sequential numbering.
- **Fields are fixed** ‚Äî all fields (`.strategy`, `.host`, `.maxInstances`, etc.) are Polyglot-defined fixed fields. `#RetriggerStrategy` is a queue configuration enforced by the Trigger Monitor, not a queue metadata field.
- **Host-based dispatch** ‚Äî `.host` binds each queue to a specific host. 1 queue = 1 host. Offloading work to another host means switching queues (e.g., via `=Q.Reassign`).
- **Active controls** ‚Äî nested `[Q]` lines within the definition set default pause/resume/kill behavior.
- **`live` fields** ‚Äî queue instances report runtime state: `pendingCount`, `activeCount`, `totalProcessed`. See [[metadata|user/concepts/metadata]].

## Trigger Branch

`%T` stores trigger definitions (`{T}`). Triggers are specialized pipeline subtypes that define event sources with IO-only bodies. Each `[T]` invocation in a pipeline creates a new trigger instance.

### Structure

```
%T:Folder.NewFiles:0
‚îú‚îÄ‚îÄ .<                      <- input ports
‚îÇ   ‚îî‚îÄ‚îÄ .path#path
‚îî‚îÄ‚îÄ .>                      <- output ports
    ‚îú‚îÄ‚îÄ .IsTriggered#bool   <- mandatory
    ‚îî‚îÄ‚îÄ .NewFiles#array:path
```

### Key Properties

- **Flexible instances** ‚Äî each trigger invocation creates `%T:Name:N` with sequential numbering, like pipelines.
- **IO via `.<`/`.>`** ‚Äî same as pipelines (not `.[{]`/`.[}]` like wrappers). Inputs are trigger configuration; outputs are trigger results.
- **Mandatory output** ‚Äî `>IsTriggered#bool` must exist on every trigger definition (compiler enforced).
- **`live` fields** ‚Äî trigger instances report runtime state: `status`, `lastFired`, `fireCount`. See [[metadata|user/concepts/metadata]].

## Permission Branch

`%_` stores named `{_}` permission objects. Unlike other branches, `%_` has **no `:{instance}` level** ‚Äî permissions are compile-time declarations with no runtime instances. Object names use `:` (flexible), but all subfields use `.` (fixed). All categories and capabilities are Polyglot-defined, not user-extensible. `%_` uses `permission_path` (not `instance_path`) ‚Äî see [[path-grammar|Path Grammar]]. See [[permissions]] for the full permission system.

### Structure

```
%_
+-- :DataCeiling                       <- named {_} permission object
|   +-- .intent                        #PermissionIntent (Ceiling | Grant)
|   +-- .target                        __PermissionTarget
|   |   +-- .category                  #PermissionCategory (File, Web, Database, System, Crypto, IPC, Device, Memory)
|   |   +-- .capability                per-category enum (#FileCapability, #WebCapability, etc.)
|   |   +-- .scope                     __PermissionScope
|   |       +-- .pattern               #GlobPattern (ceiling) or specific value (grant)
|   |       +-- .direction             #IODirection (Inbound, Outbound, Both)
|   +-- .grant                         __PermissionGrant
|   |   +-- .level                     #AccessLevel (Allow, Deny)
|   |   +-- .authority                 #GrantAuthority (Package, Pipeline)
|   |   +-- .intent                    #PermissionIntent (Ceiling, Grant)
|   +-- .resource                      __ResourceDescriptor
|   |   +-- .os                        #OSTarget (Any, Linux, Windows, MacOS)
|   |   +-- .protocol                  #Protocol (File, TCP, UDP, HTTPS, IPC, SharedMemory, USB, Bluetooth)
|   |   +-- .handle                    #HandleKind (Path, ConnectionString, Descriptor, Address, DeviceID)
|   +-- .audit                         __PermissionAudit
|       +-- .log                       #AuditLevel (None, OnUse, OnDeny, All)
|       +-- .alert                     #AlertLevel (None, OnDeny, OnEscalation)
+-- :ReportReader                      <- another named {_} object
    +-- ...
```

### Key Properties

- **Named objects** ‚Äî each `{_}` definition creates a named entry under `%_` (e.g., `%_:DataCeiling`). Object names use `:` flexible navigation.
- **Fixed subfields (`.`)** ‚Äî all fields within a permission object use `.` fixed-field navigation. The `__Permission` schema is Polyglot-defined, not user-extensible.
- **No instances** ‚Äî permissions are per-definition, resolved at compile time. No `:{instance}` level exists. No runtime metadata.
- **No `live` fields** ‚Äî all permission data is static. The compiler resolves permissions entirely during compilation.
- **Fully filled** ‚Äî every `{_}` object must have all leaf fields assigned. Empty leaves are a compile error.
- **Identifier tiers** ‚Äî `_` = permission object, `__` = permission descriptor (schema), `___` = constraint descriptor. Mirrors `#`/`##`/`###`.
- **Nested under `%@` and `%=`** ‚Äî permissions also appear as `._` subsections under package (`%@:<address>._`) and pipeline (`%=:<name>:<instance>._`) branches, representing the package ceiling and pipeline-level grant references respectively.
