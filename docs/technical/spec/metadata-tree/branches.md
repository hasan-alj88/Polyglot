# Branch Specifications

<!-- @source:metadata-tree/INDEX -->

This file specifies the pipeline, wrapper, queue, trigger, and permission branches of the metadata tree.

## Pipeline Branch

`%=` stores pipeline instances (`{=}`). Each concurrent run of the same pipeline definition creates a new instance with sequential numbering (`:0`, `:1`, `:2`). Within each instance, jobs are tracked by UID вЂ” not sequential numbers.

### Structure

```
%=:MyPipeline:0
в”њв”Ђв”Ђ .<                           <- input ports
в”‚   в””в”Ђв”Ђ .data#Serial
в”њв”Ђв”Ђ .>                           <- output ports
в”‚   в””в”Ђв”Ђ .result#Serial
в”њв”Ђв”Ђ .jobs                        <- job instances (UID-keyed)
в”‚   в”њв”Ђв”Ђ :abc123#Job              <- individual job
в”‚   в”‚   в”њв”Ђв”Ђ .PID
в”‚   в”‚   в”њв”Ђв”Ђ .status
в”‚   в”‚   в”њв”Ђв”Ђ .hierarchy           <- e.g., "MyPipeline/abc123"
в”‚   в”‚   в””в”Ђв”Ђ ...
в”‚   в””в”Ђв”Ђ :def456#Job
в”‚       в”њв”Ђв”Ђ .parent              <- "abc123" (parent job)
в”‚       в””в”Ђв”Ђ ...
в””в”Ђв”Ђ .status                      <- pipeline instance status
```

### Key Properties

- **Sequential instances** вЂ” each concurrent run creates `%=:Name:N` with sequential numbering. Instance `:0` is the first run, `:1` the second, etc.
- **UID-keyed jobs** вЂ” within each instance, `.jobs` contains jobs identified by UID (generated by the Trigger Monitor), not sequential numbers. This allows the queue system to track jobs by globally unique ID.
- **IO boundaries define jobs** вЂ” the compiler determines job start/end via pipeline IO state: inputs must be Final to start, outputs Final or Failed to complete. All actions and reactions are fully deterministic once compiled.
- **Job hierarchy** вЂ” jobs form a parent/child tree within the instance. `.hierarchy` stores the dot-separated path (e.g., `ProcessData/job1/job3`). `.parent` references the parent job UID.
- **`live` fields** вЂ” pipeline instances report runtime state: `status`, IO port values. See [[metadata|user/concepts/metadata]].

## Wrapper Branch

`%W` stores wrapper definitions (`{W}`). Wrappers provide setup/cleanup scope around pipeline execution bodies. Each `[W]` invocation in a pipeline creates a new wrapper instance.

### Structure

```
%W:DB.Connection:0
в”њв”Ђв”Ђ .[{]                     <- inputs from calling pipeline
в”‚   в””в”Ђв”Ђ .connectionString#string
в”њв”Ђв”Ђ .[}]                     <- outputs exposed to calling pipeline
в”‚   в””в”Ђв”Ђ .dbConn
в”њв”Ђв”Ђ .setup                   <- [\] setup phase
в””в”Ђв”Ђ .cleanup                 <- [/] cleanup phase
```

### Key Properties

- **Flexible instances** вЂ” each `[W]` invocation creates `%W:Name:N` with sequential numbering, like pipelines.
- **IO via `[{]`/`[}]`** вЂ” wrapper inputs (`[{]`) and outputs (`[}]`) are fixed typed data sections, analogous to `.<`/`.>` in pipelines.
- **Composite wrappers** вЂ” a `{W}` definition can contain `[W]` references to other wrappers inside `[\]` or `[/]`, creating nested wrapper instances.
- **`live` fields** вЂ” wrapper instances report runtime state: `status`, `errors`, `setupDuration`. See [[metadata|user/concepts/metadata]].

## Queue Branch

`%Q` stores queue definitions (`{Q}`). Queues manage pipeline dispatch ordering and concurrency. Each queue dispatches pipelines and tracks active/pending counts.

### Structure

```
%Q:GPUQueue:0
в”њв”Ђв”Ђ .strategy#QueueStrategy        <- FIFO, LIFO, Priority
в”њв”Ђв”Ђ .host#String                   <- target host (1 queue = 1 host)
в”њв”Ђв”Ђ .maxInstances#UnsignedInt      <- max parallel instances per pipeline
в”њв”Ђв”Ђ .maxConcurrent#UnsignedInt     <- max other pipelines alongside
в”њв”Ђв”Ђ .resourceTags#Array:ResourceTag <- resource constraint tags
в”њв”Ђв”Ђ .killPropagation#KillPropagation <- Cascade or Downgrade
в”њв”Ђв”Ђ .maxWaitTime#String            <- max time before escalation
в”њв”Ђв”Ђ .description#String            <- human-readable description
в””в”Ђв”Ђ .controls                      <- active queue controls
    в”њв”Ђв”Ђ .pause
    в”њв”Ђв”Ђ .resume
    в””в”Ђв”Ђ .kill
```

### Key Properties

- **Flexible instances** вЂ” each queue use creates `%Q:Name:N` with sequential numbering.
- **Fields are fixed** вЂ” all fields (`.strategy`, `.host`, `.maxInstances`, etc.) are Polyglot-defined fixed fields. `#RetriggerStrategy` is a queue configuration enforced by the Trigger Monitor, not a queue metadata field.
- **Host-based dispatch** вЂ” `.host` binds each queue to a specific host. 1 queue = 1 host. Offloading work to another host means switching queues (e.g., via `=Q.Reassign`).
- **Active controls** вЂ” nested `[Q]` lines within the definition set default pause/resume/kill behavior.
- **`live` fields** вЂ” queue instances report runtime state: `pendingCount`, `activeCount`, `totalProcessed`. See [[metadata|user/concepts/metadata]].

## Trigger Branch

`%T` stores trigger definitions (`{T}`). Triggers are specialized pipeline subtypes that define event sources with IO-only bodies. Each `[T]` invocation in a pipeline creates a new trigger instance.

### Structure

```
%T:Folder.NewFiles:0
в”њв”Ђв”Ђ .<                      <- input ports
в”‚   в””в”Ђв”Ђ .path#path
в””в”Ђв”Ђ .>                      <- output ports
    в”њв”Ђв”Ђ .IsTriggered#bool   <- mandatory
    в””в”Ђв”Ђ .NewFiles#array:path
```

### Key Properties

- **Flexible instances** вЂ” each trigger invocation creates `%T:Name:N` with sequential numbering, like pipelines.
- **IO via `.<`/`.>`** вЂ” same as pipelines (not `.[{]`/`.[}]` like wrappers). Inputs are trigger configuration; outputs are trigger results.
- **Mandatory output** вЂ” `>IsTriggered#bool` must exist on every trigger definition (compiler enforced).
- **`live` fields** вЂ” trigger instances report runtime state: `status`, `lastFired`, `fireCount`. See [[metadata|user/concepts/metadata]].

## Permission Branch

`%_` stores permission declarations. Unlike other branches, `%_` has **no `:{instance}` level** and **no `:` flexible fields** вЂ” permissions are compile-time declarations with an entirely fixed schema. All categories and capabilities are Polyglot-defined, not user-extensible. See [[permissions]] for the full permission system.

### Structure

```
%_
в”њв”Ђв”Ђ .File
в”‚   в”њв”Ђв”Ђ .read               #string  (glob pattern)
в”‚   в”њв”Ђв”Ђ .write              #string
в”‚   в”њв”Ђв”Ђ .execute            #string
в”‚   в””в”Ђв”Ђ .delete             #string
в”њв”Ђв”Ђ .Web
в”‚   в”њв”Ђв”Ђ .request
в”‚   в”‚   в””в”Ђв”Ђ .<              (IO inputs)
в”‚   в””в”Ђв”Ђ .socket
в”‚       в””в”Ђв”Ђ .<
в”њв”Ђв”Ђ .Database
в”‚   в”њв”Ђв”Ђ .connect
в”‚   в”‚   в””в”Ђв”Ђ .<
в”‚   в”њв”Ђв”Ђ .read               #string
в”‚   в””в”Ђв”Ђ .write              #string
в”њв”Ђв”Ђ .System
в”‚   в”њв”Ђв”Ђ .env                #string
в”‚   в”њв”Ђв”Ђ .process
в”‚   в”‚   в””в”Ђв”Ђ .<
в”‚   в””в”Ђв”Ђ .signal             #string
в”њв”Ђв”Ђ .Crypto
в”‚   в”њв”Ђв”Ђ .key, .sign, .encrypt   #string
в”њв”Ђв”Ђ .IPC
в”‚   в”њв”Ђв”Ђ .send, .receive
в”‚   в”‚   в””в”Ђв”Ђ .<
в”‚   в””в”Ђв”Ђ .subscribe          #string
в”њв”Ђв”Ђ .Device
в”‚   в”њв”Ђв”Ђ .camera, .microphone, .location, .bluetooth   #bool
в””в”Ђв”Ђ .Memory
    в”њв”Ђв”Ђ .allocate, .shared   #string
```

### Key Properties

- **All fixed (`.`)** вЂ” every level uses `.` fixed-field navigation. No `:` flexible fields anywhere in `%_`. Permission categories and capabilities are predefined by Polyglot.
- **No instances** вЂ” permissions are per-definition, resolved at compile time. No runtime metadata exists.
- **No `live` fields** вЂ” all permission data is static. The compiler resolves permissions entirely during compilation.
- **Nested under `%@` and `%=`** вЂ” permissions also appear as `._` subsections under package (`%@:<address>._`) and pipeline (`%=:<name>:<instance>._`) branches, representing the package ceiling and pipeline-level declarations respectively.
- **IO-form capabilities** вЂ” capabilities like `.request`, `.connect`, `.send` use `.<` for their IO input parameters, mirroring the IO form syntax in `[_]` declarations.
