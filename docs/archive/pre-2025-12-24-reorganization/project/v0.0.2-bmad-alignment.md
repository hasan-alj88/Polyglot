## How They Work Together

### For Lexer Implementation (Story 1.2):

**v0.0.2 defines:** All token types that MUST be recognized for the complete language
- Block markers: `[|]`, `[X]`, `[i]`, `[o]`, `[r]`, `[p]`, `[~]`, `[<]`, `[>]`, `[Y]`, `[t]`, `[Q]`, `[W]`, `[#]`, `[!]`, `[A]`, `[?]`, `[=]`, `[*]`, `[+]`, `[&]`, `[^]`, `[.]`, `[{]`, `[}]`
- Operators: `|`, `~`, `<<`, `>>`, `@`, `#`, `!`, `=?`, `>?`, `<?`, `>=?`, `<=?`, `=!?`, `>!?`, `<!?`, `>=!?`, `<=!?`, `<!?<`, `?[`, `?(`, `]`, `)`, `?*`, `?re`
- Identifiers, literals, comments, structural tokens

**BMAD defines:** Implementation scope and priorities
- **MVP (Story 1.2):** Implement ALL token types (foundation must be complete)
- **Rationale:** Lexer must tokenize any valid v0.0.2 file, even if some features won't be fully compiled until post-MVP

### For Runtime Wrappers (Epic 7):

**v0.0.2 defines:** `[W] |W.Runtime.Version` syntax for invoking runtime wrappers

**BMAD defines:** Implementation phases
- **MVP (Epic 7):** Python 3.11+ runtime wrapper only (proof-of-concept)
- **v1.0 (Post-MVP):** Node.js, Rust, Go runtime wrappers
- **Vision:** C/C++ header loading, community-contributed wrappers

### For Queue Management (Epic 5):

**v0.0.2 defines:** `[Q]` block marker and queue routing syntax

**BMAD defines:** Implementation phases
- **MVP (Epic 5):** Single dispatch queue only
- **Post-MVP:** Multiple custom queues, priority algorithms, pause/resume functionality

## Critical Distinctions Clarified

### 1. Operators vs. Block Markers

**Previous Confusion:** Story 1.2 initially conflated these concepts

**v0.0.2 Clarification:**
- **Block Markers:** Square-bracket syntax (`[X]`) that define code structure (pipelines, blocks, execution modes)
- **Operators:** Symbolic syntax (`|`, `~`, `@`) that perform operations (call pipelines, transform data, access packages)

**BMAD Alignment:** Story 1.2 now correctly distinguishes these categories in acceptance criteria

### 2. Keyword Elimination

**Previous Confusion:** Some BMAD docs referenced True/False as keywords

**v0.0.2 Clarification:** Version 0.0.2 eliminated ALL 5 keywords
- `True`/`False` → `#Boolean.True` / `#Boolean.False` (reserved enumeration)
- `Fixed` → `[=]` block marker (immutable assignment)
- `Default` → inline `<<` in `[i]` declaration
- `Exposed` → `[}]` scope output (macro-exported variables)

**BMAD Alignment:** Story 1.2 now documents NO keywords, uses enumeration-based boolean syntax

### 3. Two-Phase Parsing

**v0.0.2 Architecture:** Surface Syntax (user-written) → Canonical Syntax (validated and transformed)

**BMAD Alignment:**
- Epic 1 (Lexer & Parser) implements Phase 1 (Surface Syntax parsing)
- Epic 2 (IR Generation) implements Phase 2 transformations (macro expansion, default insertion, validation)

### 4. Package Declaration Requirement

**v0.0.2 Requirement:** Every `.pg` file MUST start with package declaration block `[@] ... [X]`

**BMAD Implementation:** To be enforced in parser validation (Story 1.5 or 1.6)

## Usage Guidelines for Developers

### When Writing Stories:

1. **Reference v0.0.2 docs** for complete syntax specification
2. **Reference BMAD architecture** for technology stack and patterns
3. **Check BMAD PRD** for MVP vs post-MVP scope
4. **Follow BMAD epics** for implementation dependencies

### When Implementing Features:

1. **v0.0.2 defines WHAT syntax looks like** (the language specification)
2. **BMAD architecture defines HOW to implement** (Rust, SQLx, tokio, etc.)
3. **BMAD stories define WHEN to implement** (Epic 1 → Epic 2 → Epic 3...)

### When in Doubt:

- **Syntax questions?** → Check v0.0.2 language documentation
- **Architecture questions?** → Check BMAD architecture.md ADRs
- **Scope questions?** → Check BMAD PRD MVP definition
- **Implementation questions?** → Check BMAD story dev notes and architecture patterns
