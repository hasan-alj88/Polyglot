### 1. ✅ Indentation vs `\~\` Markers - RESOLVED

**Severity:** ~~🔴 BLOCKER~~ → ✅ RESOLVED

**Issue:** All v0.0.3.1 examples use 3-space indentation to mark iteration scope, but the version roadmap states indentation replaces `\~\` markers in **v0.0.4, not v0.0.3.1**.

**Resolution:**
- Markers with `\` (backslash) are deprecated in v0.0.3.1
- Levels are expressed with **exactly 3 spaces**
- v0.0.3.1 uses indentation-based scoping (not v0.0.4)

**Clarified syntax:**
```polyglot
[r] ~ForEach
[~] <array << $items
[~] >item >> $item

   // ✅ Exactly 3-space indentation marks iteration scope
   [r] $processed << |Transform <input << $item
```

**Action Required:**
- Update version roadmap to reflect that indentation is a v0.0.3.1 feature
- Document: "Exactly 3 spaces" requirement in specification

### 2. ✅ Chain Pack Operator Syntax - RESOLVED

**Severity:** ~~🔴 BLOCKER~~ → ✅ RESOLVED

**Issue:** The chain pack operator uses ambiguous syntax with multiple `>` characters.

**Resolution:**
Chaining uses redirect operators to link outputs across iterations:

```polyglot
[*] >current >> >next >> >next >> >item
//  ^^^^^^^^    ^^^^^    ^^^^^    ^^^^^
//  output      redirect redirect final destination
//  name        to next  to next  (unpack output in next iteration)
```

**Grammar:**
```
pack_output ::= '>' name ('>>' '>' name)* '>>' '>' unpack_reference
```

**Example:**
```polyglot
[V] *Chain
[*] <current << $next_value
[*] >next >> >item                    // ✅ "next" redirects to unpack output "item" in next iteration
```

**Clarification:**
- First `>next` is the pack output name
- `>>` is the redirect operator
- `>item` references the unpack output `[~] >item` in the next iteration
- Can chain multiple redirects: `>> >a >> >b >> >c`

**Action Required:**
- Document redirect operator `>>` in grammar specification
- Clarify that `>name` after `>>` references unpack outputs, not pack outputs

### 3. ✅ Fire-and-Forget Scope Termination - RESOLVED

**Severity:** ~~🔴 BLOCKER~~ → ✅ RESOLVED

**Issue:** Fire-and-forget loops have no `[v]` pack marker - when does scope end?

**Resolution:**
- `[v]` is the **join marker** for collecting results
- Fire-and-forget mode (`[b]`) has **no joining/collection**
- Therefore, **no `[v]` marker needed**
- Scope ends at **dedentation** (return to loop-level indentation)

**Clarified syntax:**
```polyglot
[b] ~ForEach
[~] <array << $events
[~] >item >> $event

   [r] |LogEvent <event << $event
   [r] |SendWebhook <event << $event
   // ✅ Iteration scope ends here (dedented to loop level)

// Back to main pipeline scope (dedented to pipeline level)
[r] // Continue main pipeline
```

**Rule:** Scope termination is always by dedentation (3-space reduction)

**Action Required:**
- Document dedentation-based scope termination
- Clarify that `[v]` is optional for `[b]` mode (must be absent)

### 4. ✅ Multiple Pack Operations - RESOLVED (State-Based, Not Name-Based)

**Severity:** ~~🔴 BLOCKER~~ → ✅ RESOLVED

**Issue:** How do pack operators like `*Partition` map inputs to outputs?

**Resolution:**
Pack operators use **variable state checking**, NOT name matching!

**Clarified syntax:**
```polyglot
[v] *Partition.Status
[*] <item << $result                  // Single input (NOT named success/failure)
[*] >successes >> $results            // Collect Final states
[*] >failures >> $errors              // Collect Faulted states
```

**Key insight:** Every variable has states:
- **Final** - Operation succeeded, value available
- **Faulted** - Operation failed, has error info
- **Pending** - Awaiting completion
- **Default** - Has default value
- **Cleared** - Scope ended (terminal)

**How `*Partition.Status` works:**
1. Takes `<item` from each iteration
2. **Internally checks `$result;state`** (reserved field on ALL variables)
3. If `$result;state == Final` → adds to `>successes` output
4. If `$result;state == Faulted` → adds to `>failures` output

**No name matching needed!** The operator uses the variable state system.

**Action Required:**
- Document that `*Partition.Status` and similar operators use `$var;state` internally
- Add to standard pack operators: `*Partition.Status`
- Clarify variable state checking syntax: `$var;state =? #;Variables;States;Final`

### 5. ✅ Nested Loop Scope Visibility - RESOLVED

**Severity:** ~~🟡 HIGH~~ → ✅ RESOLVED

**Issue:** Nested loops have undefined scope interaction.

**Resolution:**
- Scope visibility is determined **through indentation** (exactly 3 spaces per level)
- Inner scopes can access outer scope variables (lexical scoping)
- Indentation creates clear scope hierarchy

**Clarified semantics:**
```polyglot
[r] ~ForEach
[~] <array << $outer_items
[~] >item >> $outer_item              // Outer scope variable

   [r] ~ForEach                       // Inner loop (3 spaces deeper)
   [~] <array << $inner_items
   [~] >item >> $inner_item           // Inner scope variable (different name - no collision)

      [r] $combined << $outer_item + $inner_item  // ✅ Can access both scopes
```

**Rules:**
1. ✅ Inner scopes can read outer iteration variables
2. ✅ Variables at different levels can have different names (no collision)
3. ✅ Nesting depth unlimited (3 spaces per level)
4. ✅ Lexical scoping (inner sees outer)

**Action Required:**
- Document lexical scoping rules in specification
- Add nested loop examples to spec

### 6. ✅ Execution Mode Mixing - RESOLVED

**Severity:** ~~🟡 HIGH~~ → ✅ RESOLVED

**Issue:** Can you mix execution modes in nested loops?

**Resolution:**
Different execution markers serve different purposes and can be combined:

**Execution Markers:**
- `[r]` - Sequential execution
- `[p]` - Parallel execution (requires `[v]` to collect)
- `[b]` - Background/fire-and-forget (no collection)
- `[f]` - Forked pipeline (conditional branching)
- `[!]` - Error handling

**Clarified semantics:**
```polyglot
[p] ~ForEach                          // ✅ Parallel outer loop
[~] <array << $outer

   [r] ~ForEach                       // ✅ Sequential inner loop (each outer iteration runs sequentially)
   [~] <array << $inner

      [r] $result << $outer + $inner  // ✅ Can access outer variables

      [v] *Join.All
      [*] <item << $result
      [*] >array >> $inner_results

   [v] *Join.All                       // Collect parallel outer results
   [*] <item << $inner_results
   [*] >array >> $all_results
```

**Rules:**
1. ✅ All mode combinations allowed
2. ✅ Each level has independent execution mode
3. ✅ Modes compose naturally based on indentation

**Action Required:**
- Document execution mode composition
- Add nested mode examples

### 7. ✅ Sequential Mode Pack Operator Requirements - RESOLVED

**Severity:** ~~🟡 HIGH~~ → ✅ RESOLVED

**Issue:** Does `[r]` sequential mode require `[v]` pack operator?

**Resolution:**
Pack operator requirements depend on **whether you want to collect results**:

**Sequential `[r]` WITH collection:**
```polyglot
[r] ~ForEach
[~] <array << $items
[~] >item >> $item
   [r] $processed << |Transform <input << $item

   [v] *Collect.Into.Array          // ✅ Pack operator for collecting
   [*] <item << $processed
   [*] >array >> $results
```

**Sequential `[r]` WITH last-only collection:**
```polyglot
[r] ~ForEach
[~] <array << $items
[~] >item >> $item
   [r] $processed << |Transform <input << $item

   [v] *Collect.Last                 // ✅ Pack operator for last item only
   [*] <item << $processed
   [*] >last >> $result
```

**Parallel `[p]` WITH collection:**
```polyglot
[p] ~ForEach
[~] <array << $items
[~] >item >> $item
   [r] $processed << |Transform <input << $item

   [v] *Collect.Into.Array           // ✅ Pack operator required
   [*] <item << $processed
   [*] >array >> $results
```

**Background `[b]` WITHOUT collection:**
```polyglot
[b] ~ForEach
[~] <array << $items
[~] >item >> $item
   [r] $processed << |Transform <input << $item

   // ✅ NO pack operator - fire-and-forget
```

**Rules:**
1. ✅ `[r]` and `[p]` REQUIRE `[v]` pack operator when collecting results
2. ✅ `[b]` MUST NOT have `[v]` pack operator (no collection)
3. ✅ If you don't want results, use `[b]` not `[r]`
4. ✅ Different pack operators: `*Collect.Into.Array`, `*Collect.Into.Set`, `*Collect.Last`

**Action Required:**
- Document pack operator requirements per execution mode
- Add examples for all three modes

### 8. ✅ Variable State Checking Semantics - RESOLVED

**Severity:** ~~🟡 HIGH~~ → ✅ RESOLVED

**Issue:** How does variable state checking work?

**Resolution:**
Complete variable state system now documented (see `docs/user/variable-state-system.md` and `docs/technical/variable-states-specification.md`)

**Variable States (6 core states):**
1. **Pending** - Declared without value, awaiting push
2. **Default** - Has default value, can override once
3. **Final** - Value available, immutable
4. **Faulted** - Operation failed, has error info
5. **Cleared** - Scope ended, memory freed (terminal)

**Plus 4 queue management states:** Retrying, Paused, Cached, Dirty

**Reserved Fields on ALL Variables:**
1. **`;state`** - Query current state
   ```polyglot
   [f] $var;state =? #;Variables;States;Final
      [r] // Use value
   [f] $var;state =? #;Variables;States;Faulted
      [r] // Handle error
   ```

2. **`;errors`** - Error information (when Faulted)
   ```polyglot
   [f] $var;state =? #;Variables;States;Faulted
      [r] |LogError <errors << $var;errors
   ```

**State Checking Syntax:**
```polyglot
// In v0.0.3.1 (uses $ prefix)
$var;state =? #;Variables;States;Final      // Check if Final
$var;state =? #;Variables;States;Faulted    // Check if Faulted
$var;state =? #;Variables;States;Pending    // Check if Pending
```

**Collection States:**
- Collections themselves have states (Final, Faulted, etc.)
- Individual elements have independent states
- Pack operators filter by element state:
  - `*Join.All` - Collects all (Final + Faulted)
  - `*Join.All.Success` - Collects only Final states
  - `*Join.All.Failures` - Collects only Faulted states
  - `*Partition.Status` - Partitions by Final vs Faulted

**Answers to original questions:**
1. ✅ `;state` is a **reserved field** on ALL variables (compiler-managed)
2. ✅ Can check state of collections: `$results;state`
3. ✅ Collections are arrays of variables, each with own state
4. ✅ `*Join.All.Success` **automatically filters** by checking `$var;state` internally
5. ✅ Variable states ARE the error system (Faulted state carries errors)

**Action Required:**
- Update v0.0.3.1 spec to use `$` prefix (not `,`)
- Document `;state` and `;errors` reserved fields
- Add examples of state checking in loops
- Document pack operators that use state filtering

### 9. ✅ While/Until Conditional Loop Semantics - RESOLVED

**Severity:** ~~🟡 HIGH~~ → ✅ RESOLVED

**Issue:** While/Until loops were drastically underspecified.

**Resolution:**
While loops still unpack arrays, but use `*Until` pack operator to check termination condition:

```polyglot
[r] ~While
[~] <array << $arr_var                // ✅ Still array-based (not infinite)
[~] >item >> $current                 // Each item from array

   [r] // Do something with $current and produce $flag
   [r] $flag << ... // some condition check

   [v] *Until                         // ✅ Conditional pack operator
   [*] <condition << $flag            // Check flag to stop

   [V] *Collect.Last                   // Collect final result
   [*] <last << $current
   [*] >main >> $var
```

**Key insights:**
1. `~While` unpacks arrays like `~ForEach`
2. Loop body produces condition variable `$flag`
3. `[v] *Until` checks condition each iteration
4. If condition met, loop terminates early
5. `[V] *Collect.Last` gets final iteration result

**Resolution - Marker Clarification:**
- ✅ Correct marker is `[v]` (lowercase) - not `[V]`
- ✅ Multiple pack operators allowed in same loop
- ✅ `[v]` is the universal join/pack marker

**Action Required:**
- Update ALL examples to use lowercase `[v]` instead of `[V]`
- Document multiple pack operator semantics
- Add `*Until` to standard pack operators list

### 10. ✅ Pack Output Type Inference - RESOLVED

**Severity:** ~~🟡 HIGH~~ → ✅ RESOLVED

**Issue:** How is pack output type determined with mixed iteration results?

**Resolution:**
- Polyglot is a **strongly typed language**
- All variables **must be declared with datatype**
- Pack operator output type comes from **variable declaration**

**Clarified semantics:**
```polyglot
[r] ~ForEach
[~] <array << $items
[~] >item >> $item

   [f] $item >? 0
      [r] $result:int << $item           // ✅ Explicit type
   [f] *?
      [r] $result:int << -1              // ✅ Must match declared type

   [v] *Join.All
   [*] <item << $result
   [*] >array >> $results:array.int      // ✅ Type declared at output
```

**Alternative with union/serial type:**
```polyglot
   [f] $item >? 0
      [r] $result:serial << $item        // ✅ Can hold any serializable type
   [f] *?
      [r] $result:serial << "negative"   // ✅ Also serial

   [v] *Join.All
   [*] <item << $result
   [*] >array >> $results:array.serial   // ✅ Array of serial
```

**Rules:**
1. ✅ All variables require explicit type declaration
2. ✅ Pack output type = variable declaration type
3. ✅ Iteration variables must conform to declared type
4. ✅ Use `:serial` for heterogeneous types

**Action Required:**
- Document type declaration requirements
- Add type examples to specification
