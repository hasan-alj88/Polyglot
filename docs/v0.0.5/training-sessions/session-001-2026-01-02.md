## Critical Corrections & Learnings

### 1. Variable Lifecycle (Immutability)

**Mistake:** Treating variables as mutable
```polyglot
[r] $status << "unknown"
[f] $condition
   [r] $status << "success"  %% ERROR: already Final!
```

**Correct:** Use Default → Final pattern
```polyglot
[r] $status:string <~ "unknown"  %% Default
[f] $condition
   [r] $status << "success"      %% Final (overrides)
```

**State Machine:**
```
Declared → Pending → [default <~] → Default → [final <<] → Final → Released
                  ↘ [final <<] → Final → Released
                  ↘ [failed] → Faulted → Released
```

**Rules:**
- Maximum 2 pushes: one default `<~`, one final `<<`
- Default is tentative, can be overridden
- Final is immutable
- Pack operations `[*]` accumulate across iterations

### 2. Inline Pipelines

**Mistake:** Mixing inline syntax with I/O wiring
```polyglot
[r] $response:serial << |HTTP.Get
 |  <url << $url
 |  >response >> $response  %% WRONG: not inline!
```

**Correct:** Pick ONE pattern
```polyglot
%% Option A: Inline (single value)
[r] $timestamp:dt << |DT.Now""

%% Option B: Full I/O wiring
[r] |HTTP.Get
 |  <url << $url
 |  >response >> $response
```

**Rule:** Inline pipelines replace ONE value/variable only.

### 3. Pipeline Structure Order (STRICT)

**Required Order:**
```
{|} → [t] → [<] → [>] → [w] → execution markers
```

**Execution markers:** `[r]`, `[p]`, `[*]`, `[b]`, `[f]`, `[s]`, `[>]`

**Mistake:** Declaring variables before wrappers
```polyglot
{|} |MyPipeline
[t] |T.Cli"start"
[r] $config << #Config  %% WRONG: before [<][>][w]
[w] |W.DB.Connect
```

**Correct:**
```polyglot
{|} |MyPipeline
[t] |T.Cli"start"
[<] <input:string
[>] >output:serial
[w] |W.DB.Connect
 |  >db >> $dbConn
[r] $config << #Config  %% OK: after [w]
```

### 4. Error Handling with `[!]`

**Concept:** `[!]` is like `[f]` for errors - creates conditional branches.

**Pattern:**
```polyglot
[r] |Pipeline
 |  <input << $value
   [!] !SpecificError
      %% Handle specific error

   [!] !AnotherError
      %% Handle another error

   [!] !*
      %% Catch-all (including success) - REQUIRED
```

**Rules:**
- Must be exhaustive (like forks)
- `[!] !*` catches all remaining cases
- Compile error if non-exhaustive
- Can nest in iterations

**Output Handling:**
```polyglot
[!] !File.Write.Error
   [>] >report
      [.] .status:error << !File.Write.Error
      [.] .message:string << "Write failed"

[!] !*
   [>] >report
      [.] .status:error << !Success
      [.] .message:string << "Success"
```

### 5. Reserved vs User-Defined Enums

**Reserved Enums:**
- Prefix: `-`
- Defined in YAML (stdlib)
- Examples: `-DT-Now`, `-DB-RDBMS-Posgresql`, `-Success`
- Usage: `-Calendar-Hijri-Standard`, `-Email-Status-Success`

**User-Defined Enums:**
- Prefix: `#`
- Defined in code with `{#}`
- Examples: `#Campaign.Status`, `#Backup.Config`
- Usage: `#Campaign.Status.Success`

**Mistake:** Using `-` prefix for user enums
```polyglot
{-} -Backup-Status  %% WRONG: reserved syntax
```

**Correct:**
```polyglot
{#} #Backup.Status  %% User-defined
[.] .Success
[.] .Failed
{x}
```

### 6. Type System - When Schema is Known

**Generic serial (less safe):**
```polyglot
[>] >summary:serial
```

**Typed schema (recommended):**
```polyglot
[>] >summary#Campaign.Report
```

**Benefits:**
- Compile-time type checking
- Auto-completion in IDEs
- Refactoring safety
- Self-documenting

**Rule:** Use enum types when structure is known at compile time.

### 7. Subfield Marker `[.]` - Universal Usage

**Consistent across:**

1. **Enum field definitions:**
```polyglot
{#} #Config
[s] << |YAML.Load"file.yaml"
   [.] .db_host:string << .connection.host
   [.] .db_port:int << .connection.port
```

2. **Output field definitions:**
```polyglot
[>] >report
   [.] .status:error << !Success
   [.] .total_errors:uint << $errorCount
```

3. **Input field definitions:**
```polyglot
[<] <request
   [.] .user_id:string
   [.] .action:string
```

**Alternative (dotted syntax):**
```polyglot
[>] >report.status:error << !Success
[>] >report.total_errors:uint << $errorCount
```

### 8. Field Naming Convention

**Mistake:** Using dashes creates ambiguity
```polyglot
[.] .total-customers:uint  %% Looks like -customers reserved enum!
```

**Correct:** Use underscores for compound names
```polyglot
[.] .total_customers:uint  %% Clear compound field name
```

**Rules:**
- `field_name` → underscore (compound user fields)
- `field-ReservedType` → dash (reserved enum reference)
- `-ReservedEnum-Value` → dashes (reserved enum paths)

### 9. DateTime Type

**Mistake:** Using full name
```polyglot
.processed_at:datetime  %% WRONG
```

**Correct:** Short form
```polyglot
.processed_at:dt  %% Correct
```

### 10. Schema-Based Wrapper Inputs

**Old (verbose):**
```polyglot
[w] |W.DB.Postgresql
 |  <db_host:string << #Config.db_host
 |  <db_port:int << #Config.db_port
 |  <db_user:string << #Config.db_user
 |  <db_password:string << #Config.db_password
```

**New (schema-based):**
```polyglot
{#} #MyApp.Database-DB-Settings
[A] #MyAppDB
[s] << |TOML.Load"\\FileDir\\config\db.toml"
   [.] .host:string << .connection.host
   [.] .port:uint << .connection.port
   [.] .username:string << .credentials.user
   [.] .password:string << .credentials.pass
   [.] .database:string << .database_name
[s][!] !*
{x}

[w] |W.DB.Postgresql
 |  <settings-DB-Settings#MyAppDB
 |  >db-DB-Connection >> $dbConn
```

**Shorthand:**
```polyglot
<settings-DB-Settings#MyAppDB
```

Equivalent to:
```polyglot
<settings-DB-Settings << -DB-Settings#MyAppDB
```

**Benefits:**
- Type safety
- Less verbose
- Self-documenting
- Reusable

### 11. Placeholder for Empty Actions

**Mistake:** Just comments
```polyglot
[!] !*
   %% Continue normally
```

**Correct:** Explicit placeholder
```polyglot
[!] !*
   [r] |U.Do.Nothing
```

### 12. Exhaustive Conditionals

**Mistake:** Non-exhaustive conditions
```polyglot
[f] $failedCount ?> 0
[&] $successCount ?> 0
   [r] $status << #Status.PartialFailure

[f] $failedCount ?= $total
   [r] $status << #Status.Failed

%% COMPILE ERROR: Not exhaustive!
```

**Correct:** Add catch-all
```polyglot
[f] $failedCount ?= $total
   [r] $status << #Status.Failed

[f] $failedCount ?> 0
[&] $failedCount ?< $total
   [r] $status << #Status.PartialFailure

[f] *?
   [r] |U.Do.Nothing  %% Or appropriate action
```

### 13. Trigger Patterns

**Simple (inline):**
```polyglot
[t] |T.Cli"start"
[t] |T.Cron"0 * * * *"
[t] |T.Interval"30s"
```

**Complex (I/O wiring):**
```polyglot
[t] |T.Folder.NewFiles
 |  <folder:path << #Config.watch_folder
 |  >files:array.path >> <newFiles
```

**Rule:** Pick inline OR I/O wiring, not both.

### 14. Trigger Output to Input Wiring

**Pattern:**
```polyglot
[t] |T.Folder.NewFiles
 |  <folder:path << #Config.folder
 |  >files:array.path >> <newFiles  %% Wire to input

[<] <newFiles:array.path  %% Declare input
[>] >report:serial
```

**Syntax:** `>trigger_output >> <pipeline_input`

## Proposed Enhancements

### 1. Code Block Marker `[c]`

**Current (verbose):**
```polyglot
[r] |DB.Query
 |  <query:string << |SQL""
[+] +"SELECT id, name"
[+] +"FROM customers"
[+] +"WHERE active = true"
```

**Proposed:**
```polyglot
[r] |DB.Query
 |  <query:string << |SQL""
[c] SELECT id, name
[c] FROM customers
[c] WHERE active = true
```

**Benefits:**
- Cleaner syntax
- Natural multi-line appearance
- Variable substitution: `{$variable}`
- Indentation preserved

### 2. Reserved Schema for Common Types

**-DB-Settings:**
```yaml
required_fields:
  - host:string
  - port:uint
  - username:string
  - password:string
  - database:string
```

**-HTTP-Client-Settings:**
```yaml
required_fields:
  - base_url:string
  - timeout_ms:uint
```

**-Email-Settings:**
```yaml
required_fields:
  - smtp_host:string
  - smtp_port:uint
  - from_address:string
```

### 3. Exhaustive Enum Matching

For enum-typed variables, compiler validates exhaustiveness without `[f] *?`:

```polyglot
[r] $status#Status <~ #Status.Success

[f] $status ?= #Status.Success
   [r] $msg << "success"

[f] $status ?= #Status.Failed
   [r] $msg << "failed"

%% Compiler checks: all enum values covered
```

## Key Takeaways

### For Language Users

1. **Variables are immutable** - use Default → Final pattern for conditional values
2. **Inline pipelines** replace single values only
3. **Error handling must be exhaustive** - always include `[!] !*`
4. **Use typed schemas** when structure is known
5. **Follow pipeline structure order** strictly
6. **Field names use underscores** not dashes

### For Language Designers

1. **Schema-based inputs** reduce verbosity significantly
2. **Code blocks `[c]`** would improve multi-line content readability
3. **Reserved schemas** provide type safety for common patterns
4. **Exhaustive enum matching** could eliminate need for catch-all in many cases
5. **Consistent `[.]` marker** creates unified syntax for hierarchical data

### For Implementation

1. **Compiler must validate:**
   - Exhaustive error handling
   - Pipeline structure order
   - Schema field requirements
   - Variable lifecycle states

2. **Runtime must handle:**
   - Automatic resource cleanup (wrappers)
   - Variable state transitions
   - Error propagation
   - Pack operation aggregation

### Example 7: Hello World Multi-Runtime Orchestration

#### Critical Learnings

##### 1. Wrapper Ordering (CRITICAL RULE)

**Mistake:** Placing wrappers after execution markers
```polyglot
[r] $config << #Config    %% Execution marker
[w] |W.RT.Python         %% WRONG: wrapper after execution!
```

**Correct:** ALL wrappers MUST precede ALL execution markers
```polyglot
{|} |HelloWorld
[t] |T.Cli"Hello-World"
[<] <config#HelloWorld.Config
[>] >report#Execution.Report

%% ALL WRAPPERS FIRST
[w] |W.RT.Python
 |  >environment-RTenv-python >> $pyEnv

[w] |W.RT.Rust
 |  >environment-RTenv-rust >> $rustEnv

%% THEN execution markers
[r] |RT.Python.Code
```

**Rule:** Pipeline structure: `{|} → [t] → [<] → [>] → [w] → [r][p][*][b][f][s]`

##### 2. Runtime Environment Parameters

**Canonical pattern:**
```polyglot
%% Wrapper
[w] |W.RT.{Language}
 |  <version:string << $versionConfig
 |  >environment-RTenv-{language} >> $env

%% Pipeline
[r] |RT.{Language}.Code
 |  <env.lang-RTenv-{language} << $env
 |  <env.vars.{var}:string << $value
 |  <code:string << |{Language}""
```

**Reserved enum naming:**
- Full: `-RT-Environment-{Language}`
- Alias: `-RTenv-{language}`
- Example: `-RTenv-python`, `-RTenv-rust`, `-RTenv-javascript`

##### 3. Environment Variables vs Keyword Arguments

**Environment variables (recommended for general code):**
```polyglot
<env.vars.log_file:string << $config.log_file
<env.vars.db_host:string << $config.db_host
```
- **Type:** ALL `:string` (stored in Shell/CMD environment)
- **Access:** Via language-specific env mechanisms
- **Use case:** General purpose code execution

**Keyword arguments (for single-function code):**
```polyglot
<kwargs.user_id:py.int << $userId
<kwargs.message:py.str << $message
<kwargs.timeout:rust.i32 << $timeoutMs
```
- **Type:** Native language types (`:py.str`, `:rust.i32`, `:js.number`)
- **Auto-conversion:** From Polyglot types to native types
- **Use case:** Code written as single callable function

**Mistake:** Mixing types or initializing empty then adding fields
```polyglot
<env.vars:serial << {:}              %% Contradictory!
<env.vars.log_file:path << $path     %% Wrong: not :string, negates above
```

**Correct:** Just set needed fields (empty by default)
```polyglot
<env.vars.log_file:string << $config.log_file
<env.vars.debug_mode:string << "true"
```

##### 4. Input Shorthand for Enum-Typed Inputs

**Mistake:** Redundant assignment
```polyglot
[<] <config#HelloWorld.Config << #Config  %% Redundant!
```

**Correct:** Value implied from `{#}` definition
```polyglot
[<] <config#HelloWorld.Config  %% Enough - value is #Config
```

**Rule:** When input type references enum with `[A]` alias, value assignment is implied.

##### 5. Boolean Reserved Enum

**Equivalence:**
- `:bool` ≡ `-Boolean` (reserved enum)

**Values:**
- `-Boolean-True` or alias `-True`
- `-Boolean-False` or alias `-False`

**Mistake:** Using language literals
```polyglot
<item.success:bool << false  %% WRONG: lowercase literal
<item.active:bool << true    %% WRONG: lowercase literal
```

**Correct:** Use reserved enum values
```polyglot
<item.success:bool << -False
<item.active:bool << -True
```

##### 6. Serial Construction - Dot Notation (Recommended)

**Three valid patterns:**

**Pattern A: Dot notation (recommended)**
```polyglot
[*] *Into.Array
 *  <item.runtime:string << "Python"
 *  <item.success:bool << -True
 *  <item.timestamp:dt << |DT.Now""
 *  >array >> $results
```

**Pattern B: Subfield markers**
```polyglot
[*] *Into.Array
 *  <item:serial
    [.] .runtime:string << "Python"
    [.] .success:bool << -True
 *  >array >> $results
```

**Pattern C: Inline with `[+]`**
```polyglot
[*] *Into.Array
 *  <item << {
    [+] .runtime: "Python",
    [+] .success: -True
    [+] }
 *  >array >> $results
```

**Recommendation:** Use Pattern A (dot notation) for clarity and consistency.

##### 7. Hard Syntax Rule

**Universal rule:** `<indent(s)><Marker><One-expression>` per line

**Valid:**
```polyglot
[r] $value << 42
[r] |Pipeline
 |  <input << $data
```

**Invalid:**
```polyglot
[r] $value << 42; [r] $other << 43  %% Multiple expressions!
```

       *  <item.success:bool << -True
       *  <item.message:string << "Success"
       *  <item.timestamp:dt << |DT.Now""
       *  >array >> $runtimeResults

%% Generate Report
[r] $totalRuntimes << 3

[r] $runtimeResults >> ~ForEach.Array
   [b] $current.success ?= -True
      [*] *Aggregate.Sum
       *  <value:uint << 1
       *  >sum:uint >> $successful

   [b] $current.success ?= -False
      [*] *Aggregate.Sum
       *  <value:uint << 1
       *  >sum:uint >> $failed

[r] $successful:uint <~ 0
[r] $failed:uint <~ 0

[>] >report
   [.] .total_runtimes << $totalRuntimes
   [.] .successful << $successful
   [.] .failed << $failed
   [.] .results << $runtimeResults
{x}
