# Polyglot v0.0.5 - Improvement Proposals

**Date:** 2025-12-11
**Status:** 🔬 BRAINSTORMING - For Discussion
**Based on:** v0.0.4 design principles

---

## Design Constraints

All proposals must maintain:
- ✅ No keywords (only markers)
- ✅ One line = one marker + one expression
- ✅ Indentation for nesting
- ✅ Universal hierarchy pattern (PREFIX.hierarchy)
- ✅ Explicit over implicit
- ✅ Greppability and tooling support

---

## Category 1: String & Comment Enhancements

### 1.1 Multi-line Strings

**Problem:** SQL, JSON, documentation need multiple lines.

**Current workaround:**
```polyglot
[r] ,query << "SELECT * FROM users WHERE active = true"
[r] ,line2 << " AND created > '2024-01-01'"
[r] ,full_query << ,query            // Concatenation needed
```

**Proposal A: Triple-quote with dedent**
```polyglot
[r] ,query << """
   SELECT * FROM users
   WHERE active = true
     AND created > '2024-01-01'
   """
```

**Rules:**
- Opening `"""` on separate line (or inline for single-line)
- Closing `"""` determines indentation baseline
- Content dedented relative to closing `"""`
- First newline after opening `"""` is stripped

**Proposal B: Heredoc-style with marker**
```polyglot
[r] ,query << <<SQL
   SELECT * FROM users
   WHERE active = true
     AND created > '2024-01-01'
SQL
```

**Recommendation:** Triple-quote (Proposal A) - more familiar, cleaner syntax.

---

### 1.2 Block Comments

**Problem:** Need to comment out large sections or write documentation blocks.

**Current:**
```polyglot
// Line 1 of comment
// Line 2 of comment
// Line 3 of comment
```

**Proposal: C-style block comments**
```polyglot
/*
 * Multi-line comment
 * Can span many lines
 * Good for documentation
 */
[r] ,value << 42
```

**Rules:**
- `/* ... */` for block comments
- Can nest: `/* outer /* inner */ outer */`
- Cannot appear mid-line (maintains one-marker-per-line rule)

**Recommendation:** Approve - standard, familiar, useful.

## Category 2: Collection Literals & Operations

### 2.1 Array Literals

**Problem:** Creating inline arrays is verbose.

**Current (implied):**
```polyglot
[r] ,tags:pg.array.pg.string << :pg.array.pg.string
[r] ,tags << |Array.Append <item << "tag1" <array << ,tags
[r] ,tags << |Array.Append <item << "tag2" <array << ,tags
```

**Proposal: Array literal syntax**
```polyglot
[r] ,tags << [: "tag1", "tag2", "tag3" :]
[r] ,numbers << [: 1, 2, 3, 4, 5 :]
[r] ,mixed << [: ,var1, "literal", 42, ,var2 :]
```

**Syntax:** `[: element1, element2, ... :]`
- Uses `[:` and `:]` brackets (consistent with marker style)
- Comma-separated elements
- Type inferred from elements or explicit annotation
- Empty array: `[: :]` with type annotation

**Examples:**
```polyglot
[r] ,empty:array.string << [: :]
[r] ,tags << [: "go", "rust", "polyglot" :]
[r] ,mixed << [: ,user.name, ,user.email :]
```

---

### 2.2 Struct Literals Shorthand

**Problem:** Repeating field names when variable matches.

**Current:**
```polyglot
[r] ,person << #Person
   [.] name << ,name
   [.] email << ,email
   [.] age << ,age
```

**Proposal: Shorthand when variable name matches field**
```polyglot
[r] ,person << #Person
   [.] name                            // Implies << ,name
   [.] email                           // Implies << ,email
   [.] age                             // Implies << ,age
   [.] city << ,location.city          // Explicit when different
```

**Rule:** If no `<<` operator, assumes `<< ,fieldname`

**Recommendation:** Approve - common pattern, saves repetition.

---

### 2.3 Spread Operator for Arrays

**Problem:** Composing arrays, merging collections.

**Proposal: Spread operator `...`**
```polyglot
[r] ,base_tags << [: "web", "api" :]
[r] ,all_tags << [: ...,base_tags, "v2", "beta" :]
// Result: ["web", "api", "v2", "beta"]

[r] ,array1 << [: 1, 2, 3 :]
[r] ,array2 << [: 4, 5, 6 :]
[r] ,combined << [: ...,array1, ...,array2 :]
// Result: [1, 2, 3, 4, 5, 6]
```

**Syntax:** `...,variable` spreads array elements inline.

**Recommendation:** Approve - very useful for array composition.

## Category 3: Optional/Null Handling

### 3.1 Optional Types

**Problem:** Need to represent absence of value.

**Proposal: Optional type wrapper**
```polyglot
// Type definition
:pg.optional.pg.string                 // Optional string
:pg.optional.pg.int                    // Optional int

// Aliases (with @PgTypes)
:optional.string
:optional.int

// Values
:pg.optional.None                      // No value
:pg.optional.Some                      // Has value (requires wrapping)
```

**Usage:**
```polyglot
[r] ,email:optional.string << :pg.optional.None

[y] ,user_input =? ""
   [r] ,email << :pg.optional.None
[y] *?
   [r] ,email << |Optional.Some <value << ,user_input
```

---

### 3.2 Null-safe Navigation

**Problem:** Accessing optional values safely.

**Proposal: `?.` operator for null-safe field access**
```polyglot
[r] ,name << ,user?.profile?.name     // Returns optional.string
[r] ,city << ,user?.address?.city     // Returns None if any step is None
```

**Returns:** `:pg.optional.T` where T is the final field type.

**Alternative: `!!` unwrap operator**
```polyglot
[r] ,name << ,user?.name !! "Unknown"  // Provide default if None
```

**Recommendation:** Consider - very useful pattern, but adds operator complexity.

## Category 4: Range Expressions

### 4.1 Range Syntax

**Problem:** Validation ranges, iteration ranges.

**Proposal: Range operator `..`**
```polyglot
// Inclusive range
[y] ,age in 18..65                     // 18 <= age <= 65
   [r] ,eligible << #True

// Exclusive end range
[y] ,index in 0..<,length              // 0 <= index < length
   [r] ,valid << #True

// Open-ended ranges
[y] ,score in 90..                     // score >= 90
[y] ,temp in ..<32                     // temp < 32
```

**With ForEach:**
```polyglot
[p] ~ForEach.Range
   range << 1..10
   index >> ,i

   [r] ,squared << ,i * ,i
```

**Recommendation:** Approve - common pattern, very readable.

## Category 5: Destructuring

### 5.1 Tuple Destructuring for Multi-Output Pipelines

**Problem:** Multi-output pipelines require multiple lines.

**Current:**
```polyglot
[r] |FetchUserData
   <user_id << ,id
   >user >> ,user
   >metadata >> ,meta
   >timestamp >> ,ts
```

**Proposal: Tuple destructuring**
```polyglot
[r] (,user, ,meta, ,ts) << |FetchUserData
   <user_id << ,id
```

**Or inline:**
```polyglot
[r] (,user, ,meta, ,ts) << |FetchUserData <user_id << ,id
```

**With wildcards:**
```polyglot
[r] (,user, _, ,ts) << |FetchUserData <user_id << ,id
// Ignore metadata output
```

**Recommendation:** Approve - cleaner for common multi-output case.

---

### 5.2 Pattern Matching Destructuring

**Problem:** Extracting values from match patterns.

**Proposal: Destructure in match cases**
```polyglot
{#} #Result
[.] Success
   [.] data:pg.serial
[.] Error
   [.] message:pg.string
{x}

[m] outcome << result
   [?] #Result.Success{,data} ? ,data
   [?] #Result.Error{,msg} ? ""
```

**Alternative syntax:**
```polyglot
[m] outcome << result
   [?] #Result.Success >> ,data        // Extract data field
   [?] #Result.Error >> ,msg            // Extract message field
```

**Recommendation:** Consider - powerful but adds complexity.

## Category 6: Lambda/Inline Operations

### 6.1 Inline Transformations

**Problem:** Simple transformations require full pipeline definitions.

**Proposal: Inline lambda syntax with `=>` (pipeline arrow)**
```polyglot
// Map operation
[r] ,doubled << ,numbers ~> (,x => ,x * 2)

// Filter operation
[r] ,adults << ,users ~| (,u => ,u.age >? 18)

// Reduce operation
[r] ,sum << ,numbers ~< (,acc, ,x => ,acc + ,x)
```

**Operators:**
- `~>` = map (transform each element)
- `~|` = filter (keep matching elements)
- `~<` = reduce (fold/accumulate)

**Syntax:** `(,param => expression)`

**Multi-parameter:**
```polyglot
[r] ,indexed << ,items ~> ((,i, ,item) => ,item.name)
```

**Recommendation:** Consider - very powerful, but may conflict with no-keywords rule.

## Category 7: Type Inference

### 7.1 Infer Variable Types from Literals

**Problem:** Redundant type annotations for obvious cases.

**Current:**
```polyglot
[r] ,count:pg.int << 42
[r] ,name:pg.string << "Alice"
[r] ,active:pg.bool << #True
```

**Proposal: Infer from literal**
```polyglot
[r] ,count << 42                       // Infer :pg.int
[r] ,name << "Alice"                   // Infer :pg.string
[r] ,active << #True                   // Infer :pg.bool
[r] ,tags << [: "a", "b" :]            // Infer :pg.array.pg.string
```

**When explicit type required:**
- Public API boundaries (inputs/outputs)
- When type differs from literal (e.g., `:pg.int` vs `:MyType.UserId`)
- When compiler can't infer

**Recommendation:** Approve - reduces boilerplate while maintaining type safety.

---

### 7.2 Infer from Pipeline Output

**Proposal: Infer variable type from pipeline output signature**
```polyglot
// Pipeline signature known
{|} |GetUser
[>] user:pg.serial
{x}

// Variable type inferred
[r] ,user << |GetUser <id << ,user_id  // Type inferred as :pg.serial
```

**Recommendation:** Approve - obvious case, reduces redundancy.

## Category 8: Error Handling

### 8.1 Try-Catch Pattern

**Problem:** No structured error handling for pipelines that may fail.

**Proposal: `[z]` (try) marker with error binding**
```polyglot
[z] ,result << |RiskyOperation
   <input << ,data
   >output >> ,result
   !error >> ,err                      // Bind error if thrown

// Handle error
[y] ,err =? :pg.optional.None
   // Success path
   [r] ,processed << ,result
[y] *?
   // Error path
   [r] ,processed << :pg.serial
   [r] |LogError <error << ,err
```

**Alternative: Explicit error output**
```polyglot
[r] (,result, ,error) << |RiskyOperation <input << ,data

[y] ,error =? :pg.optional.None
   [r] ,success << #True
[y] *?
   [r] ,success << #False
```

**Recommendation:** Discuss - need standard error handling pattern.

## Category 9: Metadata/Attributes

### 9.1 Documentation Attributes

**Problem:** No standard way to document pipelines, types, fields.

**Proposal: `[@...]` attribute syntax**
```polyglot
[@doc "Fetches user data from database"]
[@param user_id "The unique user identifier"]
[@returns "User object with profile data"]
[@throws !Database.Connection.Timeout]
{|} |FetchUser
[<] ,user_id:string
[>] ,user:serial
{x}

[@deprecated "Use |FetchUserV2 instead"]
{|} |FetchUser
...
{x}

[@doc "User's email address"]
{#} #User
[.] [@required] email:pg.string
[.] [@optional] phone:pg.string
{x}
```

**Standard attributes:**
- `[@doc "..."]` - Documentation
- `[@deprecated "..."]` - Mark as deprecated
- `[@param name "..."]` - Parameter documentation
- `[@returns "..."]` - Return value documentation
- `[@throws !ErrorType]` - Possible errors
- `[@required]` - Field is required
- `[@optional]` - Field is optional

**Recommendation:** Approve - critical for documentation generation and IDE support.

## Category 10: Control Flow Enhancements

### 10.1 Early Return

**Problem:** Deep nesting for validation chains.

**Current:**
```polyglot
[y] ,input =? ""
   [r] ~!InvalidInput <message << "Empty input"

[y] ,input.length <? 3
   [r] ~!InvalidInput <message << "Too short"

[y] ,input.length >? 100
   [r] ~!InvalidInput <message << "Too long"

// Main logic here
```

**Proposal: `[<-]` early return marker**
```polyglot
[y] ,input =? ""
   [<-] ~!InvalidInput <message << "Empty input"

[y] ,input.length <? 3
   [<-] ~!InvalidInput <message << "Too short"

[y] ,input.length >? 100
   [<-] ~!InvalidInput <message << "Too long"

// Main logic here (not nested)
```

**Recommendation:** Consider - very useful but changes control flow semantics.

---

### 10.2 Switch/Match on Multiple Values

**Problem:** Matching complex conditions.

**Proposal: Multi-value match**
```polyglot
[m] (result, status) << (,http_code, ,body)
   [?] (200, ,b) ? |ParseSuccess <body << ,b
   [?] (404, _) ? :pg.optional.None
   [?] (500, ,b) ? ~!ServerError <response << ,b
   [?] (_, _) ? ~!UnknownError
```

**Recommendation:** Consider - powerful but complex.

## Category 11: Performance Hints

### 11.1 Parallel Execution Hint

**Problem:** No way to indicate parallelizable operations.

**Proposal: `[p||]` parallel marker variant**
```polyglot
// Sequential (current)
[p] ~ForEach
   array << ,items
   item >> ,item
   [r] |ProcessItem <data << ,item

// Parallel hint
[p||] ~ForEach
   array << ,items
   item >> ,item
   [r] |ProcessItem <data << ,item
```

**Compiler:** Can parallelize if safe, otherwise sequential fallback.

**Recommendation:** Consider - useful for performance-critical code.

---

### 11.2 Caching/Memoization Hint

**Proposal: `[@cache]` attribute**
```polyglot
[@cache ttl:300]                       // Cache for 300 seconds
{|} |ExpensiveComputation
[<] ,input:string
[>] ,result:serial
{x}
```

**Recommendation:** Consider - runtime optimization, not syntax change.

---

## Category 12: Module System

### 12.1 Explicit Exports

**Problem:** What is public vs private in a file?

**Proposal: `[X]` export marker**
```polyglot
{@} @MyApp::Utils:1.0.0.0
[A] @Utils
{x}

// Public (exported)
[X] {|} |PublicPipeline
...
{x}

// Private (not exported)
{|} |InternalHelper
...
{x}

// Public type
[X] {#} #PublicType
...
{x}
```

**Rule:** Only `[X]`-marked items visible to importers.

**Recommendation:** Approve - critical for encapsulation.
