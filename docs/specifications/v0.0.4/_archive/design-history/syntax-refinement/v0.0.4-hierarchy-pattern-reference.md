# Polyglot v0.0.4 - The Universal Hierarchy Pattern

**Core Principle:** All constructs follow `PREFIX identifier.hierarchy.path`

---

## Quick Reference Card

### All Prefixes Use Dot for Hierarchy

| Prefix | Construct | Simple | With Hierarchy |
|--------|-----------|--------|----------------|
| `,` | Variable/Parameter | `,user` | `,user.profile.name` |
| `#` | Type/Enum/Struct | `#Order` | `#OrderStatus.Processing` |
| `\|` | Pipeline | `\|Validate` | `\|Data.Transform.JSON` |
| `!` | Error | `!Timeout` | `!Network.HTTP.Timeout` |
| `@` | Registry | `@App` | `@Company::Product:1.0.0` |
| `:` | Type Path | `:pg.int` | `:pg.array.pg.string` |

---

## Pattern Breakdown

### Variables: `,identifier.field.subfield`

```polyglot
[r] ,order:pg.serial << #Order
[r] ,total << ,order.items.sum          // ,order = var, .items.sum = hierarchy
[r] ,customer_name << ,order.customer.name
[y] ,order.shipping.priority =? "express"
```

**Pattern:** `,variable` starts it, `.field` navigates it

---

### Types: `:namespace.type.parameter`

```polyglot
[r] ,count:pg.int << 0
[r] ,timestamp:pg.dt << DT.Now""
[r] ,items:pg.array.pg.string          // Array of strings (full hierarchy)
[r] ,data:pg.array.pg.serial           // Array of serialized objects
[r] ,is_valid:pg.bool << :pg.bool.true // Type prefix : with hierarchy
```

**Pattern:** `:` is the type prefix, dot navigates type hierarchy

**Important:**
- ✅ Full hierarchy: `:pg.array.pg.string`
- ❌ No curly braces: `pg.array{pg.string}` (deprecated syntax)
- ❌ No nested arrays: Use `:pg.array.pg.serial` for complex nested data

---

### Enums/Structs: `#Type.Value.Member`

```polyglot
{#} #OrderStatus
[.] Pending
[.] Processing
[.] Shipped
{x}

[r] ,status << #OrderStatus.Pending
[r] ,next_status << #OrderStatus.Processing

// Nested enum access
{#} #Result
[.] Success
   [.] OK
   [.] Created
[.] Error
   [.] BadRequest
{x}

[r] ,result << #Result.Success.OK
```

**Pattern:** `#Type` defines it, `.Value` accesses it, `.Member` drills deeper

---

### Pipelines: `|Namespace.Pipeline.Variant`

```polyglot
[r] |Data.Validate.Schema              // Namespace hierarchy
   <input:pg.string << ,data
   >valid:pg.bool >> ,is_valid

[r] |HTTP.Client.GET                   // Organized by capability
   <url:pg.url << ,endpoint
   >response:pg.serial >> ,result

[r] |Database.Users.FindByEmail        // Domain hierarchy
   <email:pg.string << ,user_email
   >user:pg.serial >> ,user_data
```

**Pattern:** `|Category.Subcategory.Pipeline` organizes pipelines into logical namespaces

---

### Errors: `!Category.Domain.Error`

```polyglot
[r] ~!Validation.Schema.InvalidJSON
   <message:pg.string << "Invalid JSON structure"
   <field:pg.string << ,field_name

[r] ~!Network.HTTP.Timeout
   <url:pg.url << ,endpoint
   <timeout:pg.int << ,max_wait

[r] ~!Database.Connection.PoolExhausted
   <pool_size:pg.int << ,max_connections
```

**Pattern:** `!Category.Domain.SpecificError` creates error taxonomy

---

### Registries: `@Namespace::Product:Version`

```polyglot
{@} @Local::MyApp:1.0.0.0
{@} @Company::ProductName:2.1.3.0
{@} @Organization::Team::Service:1.0.0.0

[A] @App
[A] @Prod
[A] @OrgService
```

**Pattern:** Double colon `::` separates namespace levels, colon `:` precedes version

## The System in One Example

```polyglot
{@} @Local::OrderService:1.0.0.0       // Registry hierarchy
[A] @OrderSvc
{x}

{#} #OrderStatus                       // Type definition
[.] Pending
[.] Processing
{x}

{|} |Orders.Process.Express            // Pipeline hierarchy
[<] ,order_id:pg.string                // Variable with type path

[t] |T.Call
[W] |W.Polyglot.Scope

// Fetch order via pipeline hierarchy
[r] |Database.Orders.FindById          // |Namespace.Domain.Action
   <id:pg.string << ,order_id
   >order:pg.serial >> ,order_data

// Access variable hierarchy
[r] ,customer_email << ,order_data.customer.email

// Match with variable hierarchy
[y] ,order_data.items.count >? 10
   [r] ,status << #OrderStatus.Processing    // Enum hierarchy

// Error with taxonomy hierarchy
[y] ,order_data.total <? 0
   [r] ~!Validation.Order.NegativeTotal      // Error hierarchy
      <order_id:pg.string << ,order_id

[>] ,status:#OrderStatus
{x}
```

**Every identifier uses the pattern:** Unique prefix declares type, dot navigates hierarchy.

**Type hierarchy examples:**
```polyglot
// Primitive types
[r] ,name:pg.string << "Alice"
[r] ,count:pg.int << 42
[r] ,active:pg.bool << :pg.bool.true

// Collection types (full hierarchy, no curly braces)
[r] ,tags:pg.array.pg.string         // Array of strings
[r] ,users:pg.array.pg.serial        // Array of serialized objects
[r] ,errors:pg.array.pg.error        // Array of error objects

// Complex nested data (use pg.serial, no nested arrays)
[r] ,data:pg.array.pg.serial         // NOT :pg.array.pg.array.T
```

## Why This Matters

### 1. Greppability
```bash
grep ",order"          # Find all order variable uses
grep "|Database"       # Find all database pipeline calls
grep "#OrderStatus"    # Find all OrderStatus enum uses
grep "!Validation"     # Find all validation errors
```

### 2. Syntax Highlighting
Simple rule: Color by prefix, dim the dots
- `,variable` → blue
- `#Type` → green
- `|Pipeline` → purple
- `!Error` → red
- `.hierarchy` → gray

### 3. Mental Model
One pattern, infinite application:
1. See the prefix → Know what it is
2. See the dot → Know it's navigating hierarchy
3. No special cases, no context needed

### 4. Composability
Hierarchies compose without ambiguity:
```polyglot
,order.items.first.product.category.name
└─┬─┘ └──────────┬──────────────────┘
  var        hierarchy (5 levels)

#Result.Success.WithWarnings.Code
└──┬──┘ └──────────┬────────────┘
  enum        hierarchy (3 levels)
```

## Migration From v0.0.3

**Single change:** `.variable` → `,variable`

**Everything else stays the same:**
- `#Type` → `#Type` ✓
- `|Pipeline` → `|Pipeline` ✓
- `!Error` → `!Error` ✓
- `pg.string` → `pg.string` ✓
- Field access `.field` → `.field` ✓

**The difference:** Now `.` has ONE job (hierarchy), and `,` is the variable prefix.

---

## Summary

**Before v0.0.4:**
- `.` did two jobs: variable prefix AND hierarchy separator
- Ambiguous: `.user.name` (which dot is which?)

**After v0.0.4:**
- `,` is the variable prefix
- `.` is ONLY for hierarchy
- Unambiguous: `,user.name` (comma=var, dot=hierarchy)
- Universal: ALL constructs use dot for hierarchy

**The Pattern:**
```
PREFIX identifier.hierarchy.path.to.value
```

**The Philosophy:**
> In Polyglot, all constructs are serialized in hierarchies.

## Alias System - Best of Both Worlds

**Problem:** Full hierarchies are verbose for common cases.

**Solution:** Import aliases from standard library!

```polyglot
{@} @Local::MyApp:1.0.0.0
[A] @App
[<] @PgTypes                           // Import standard type aliases
{x}

// Now you can use short forms:
[r] ,name:string                       // Instead of :pg.string
[r] ,count:int                         // Instead of :pg.int
[r] ,items:array.string                // Instead of :pg.array.pg.string
[r] ,valid:bool << #True               // Instead of #Boolean.True

// Full forms STILL WORK:
[r] ,backup:pg.array.pg.error          // Explicit form always valid
```

**See `v0.0.4-alias-system.md` for complete details.**
