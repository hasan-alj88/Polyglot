## Two Critical Syntax Changes

### 1. Match Operator: Simple `?` Separator

**Decision:** Use single `?` to separate pattern from result value.

**Syntax:**
```polyglot
[m] variable << source
   [?] pattern ? result
   [?] pattern ? result
   [?] * ? default_result
```

**Example:**
```polyglot
{|} |Pipeline.StatusColorMapper
[<] ,status:pg.string

[t] |T.Call
[W] |W.Polyglot.Scope

[m] color << status
   [?] "pending" ? "yellow"
   [?] "in_progress" ? "blue"
   [?] "complete" ? "green"
   [?] "failed" ? "red"
   [?] * ? "gray"

[>] ,color:pg.string
{x}
```

**Rationale:**
- ‚úÖ **Simple and clean**: Single `?` is minimal and clear
- ‚úÖ **Familiar semantics**: `?` suggests conditional/question
- ‚úÖ **Reads naturally**: "when pending, return yellow"
- ‚úÖ **Consistent with language**: Already use `?` for conditionals (`>?`, `=?`, etc.)
- ‚úÖ **No shift required**: Easy to type

**Rejected alternatives:**
- `<<?` - Too busy, awkward to type
- `<:` - Less intuitive, colon meaning unclear
- `~<` - Tilde not obviously related to matching
- `=>` - Too similar to arrow functions in other languages

### 2. Variable Prefix: Comma `,` Instead of Dot `.`

**Decision:** Use `,` for variable/parameter prefix, reserve `.` exclusively for hierarchy.

**Syntax:**
```polyglot
// Variable declaration
[r] ,variable:type << value

// Variable reference
,variable

// Hierarchy access
,variable.field.subfield
```

**Visual Comparison:**

#### Before (v0.0.3):
```polyglot
[r] .user << .database.user          // Ambiguous: which dots are prefixes?
[r] .name << .user.profile.name      // Hard to parse
[r] .total << .order.items.sum       // Where does variable end?
[y] .config.timeout >? 30
   [r] .message << "Value: {.config.timeout}"
```

#### After (v0.0.4):
```polyglot
[r] ,user << ,database.user          // Clear: comma=variable, dot=field
[r] ,name << ,user.profile.name      // Easy to parse
[r] ,total << ,order.items.sum       // Obviously ,order then .items.sum
[y] ,config.timeout >? 30
   [r] ,message << "Value: {,config.timeout}"
```

**Rationale:**
- ‚úÖ **Perfect visual distinction**: Comma=variable, dot=hierarchy separator
- ‚úÖ **Best ergonomics**: No shift key required (beats `$`)
- ‚úÖ **Easy to grep**: `,variable` instantly identifiable
- ‚úÖ **Completes the unified pattern**: All constructs now follow prefix + hierarchy
- ‚úÖ **Timing**: v0.0.4 is pre-1.0, perfect time for breaking change

**Deeper Design Principle - The Universal Hierarchy Pattern:**

In Polyglot, **ALL constructs are serialized in hierarchies** using dot `.` as the universal separator:

| Prefix | Purpose | Hierarchy Example | Meaning |
|--------|---------|-------------------|---------|
| `,` | Variable/parameter | `,order.items.total` | Variable `order`, field `items`, field `total` |
| `:` | Type path | `:pg.array.pg.string` | Namespace `pg`, type `array`, element type `string` |
| `#` | Enum/struct | `#OrderStatus.Processing` | Type `OrderStatus`, value `Processing` |
| `!` | Error | `!Network.HTTP.Timeout` | Category `Network`, domain `HTTP`, error `Timeout` |
| `\|` | Pipeline | `\|Data.Transform.JSON` | Namespace `Data.Transform`, pipeline `JSON` |

**The pattern:** `PREFIX` + `identifier` + `.hierarchy.path`

This creates perfect consistency:
- Every construct has a **unique prefix** (identifies what it is)
- Every construct uses **dot for hierarchy** (navigates structure)
- No ambiguity, no special cases
- Greppable: Search `,variable` or `\|Pipeline` or `#Type`

**Trade-offs accepted:**
- ‚ùå Unusual (comma rarely used as identifier prefix)
- ‚ùå Major breaking change (but no users yet)
- ‚öÝÔ∏è Initial cognitive adjustment (but becomes natural quickly)

**Rejected alternatives:**
- `.` (current) - Visually ambiguous with hierarchy
- `$` - Requires shift key, less ergonomic
- Other symbols unavailable or less suitable

## Complete Feature Example

Here's a real-world example showcasing both changes:

```polyglot
{|} |Pipeline.ProcessOrder
[<] ,order_id:pg.string
[<] ,priority:pg.string

[t] |T.Call
[W] |W.Polyglot.Scope

// 1. Fetch data with composition
[r] |FetchOrder
   <order_id:pg.string << ,order_id
   |> |EnrichWithCustomerData
   |> |CalculateTotals
   >order:pg.serial >> ,order_data
   >customer:pg.serial >> ,customer_data

// 2. Determine processing type using match
[m] processing_type << priority
   [?] "high" ? "express"          // Simple ? operator
   [?] "medium" ? "standard"
   [?] "low" ? "economy"
   [?] * ? "standard"

// 3. Conditional logic with hierarchy access
[y] ,order_data.total >? 1000      // ,order_data is variable, .total is field
[&] ,processing_type =? "express"
   [r] ,shipping_method << "overnight"
   [r] ,insurance << true

[y] ,order_data.total >? 100
[|] ,processing_type =? "express"
   [r] ,shipping_method << "2-day"
   [r] ,insurance << false

[y] *?
   [r] ,shipping_method << "standard"
   [r] ,insurance << false

// 4. Create struct literal
[r] ,final_order << #Order
   [.] order_id << ,order_id
   [.] customer_id << ,customer_data.id    // Field access with .id
   [.] status << #OrderStatus.Processing   // Enum value with .Processing
   [.] total << ,order_data.total          // Hierarchy access

[>] ,tracking_number:pg.string
[>] ,final_order:#Order
{x}
```

**Key observations:**
1. **Match is clean**: `[?] "high" ? "express"` reads naturally
2. **Variables are clear**: `,order_data` is obviously a variable
3. **Hierarchy is clear**: `.total` is obviously field access
4. **No ambiguity**: `,order_data.total` - comma marks the variable, dot marks the path
5. **Universal pattern**: Every construct (`,`, `#`, `!`, `|`) uses dot for hierarchy

## The Universal Hierarchy Pattern in Action

Here's a complete example demonstrating how **every prefix uses dot for hierarchy**:

```polyglot
{@} @Local::DataProcessing:1.0.0.0
[A] @DataProc
{x}

{#} #Result
[.] success:pg.bool
[.] data:pg.serial
[.] timestamp:pg.dt
{x}

{#} #ErrorSeverity
[.] Warning
[.] Error
[.] Critical
{x}

{|} |Data.Transform.JSON
[<] ,input:pg.string
[<] ,config:pg.serial

[t] |T.Call
[W] |W.Polyglot.Scope

// Variable hierarchy: ,variable.field.subfield
[r] ,database_config << ,config.database.settings
[r] ,timeout << ,database_config.connection.timeout
[r] ,max_retries << ,database_config.connection.retries

// Type hierarchy: namespace.type
[r] ,timestamp:pg.dt << DT.Now""
[r] ,is_valid:pg.bool << pg.bool.false

// Enum/Struct hierarchy: #Type.Value.Subvalue
[r] ,severity << #ErrorSeverity.Warning
[y] ,timeout <? 1000
   [r] ,severity << #ErrorSeverity.Critical

// Pipeline hierarchy: |Namespace.Pipeline.Name
[r] |Data.Validate.Schema
   <input:pg.string << ,input
   >valid:pg.bool >> ,is_valid

// Error hierarchy: !Category.Domain.Error
[y] ,is_valid =? pg.bool.false
   [r] ~!Validation.Schema.InvalidJSON
      <message:pg.string << "Invalid JSON in input"
      <severity:#ErrorSeverity << ,severity

// Result with nested hierarchy access
[r] ,result << #Result
   [.] success << ,is_valid
   [.] data << ,input
   [.] timestamp << ,timestamp

[>] ,result:#Result
{x}
```

**Every line demonstrates the pattern:**
- `,database_config.connection.timeout` - Variable prefix `,` + field hierarchy `.connection.timeout`
- `pg.dt`, `pg.bool.false` - Type prefix `:` (implicit) + type hierarchy
- `#ErrorSeverity.Critical` - Enum prefix `#` + value hierarchy
- `|Data.Validate.Schema` - Pipeline prefix `|` + namespace hierarchy
- `!Validation.Schema.InvalidJSON` - Error prefix `!` + taxonomy hierarchy

**The genius:** Once you know the prefix, you know what it is. The dot always means "navigate deeper."

## The Philosophical Core of v0.0.4

**The Unified Hierarchy Principle:**

> In Polyglot, all constructs are serialized in hierarchies. Every identifier has a unique prefix that declares its type, and every hierarchy uses dot as the universal navigator.

**The Pattern:**
```
PREFIX identifier.hierarchy.path.to.value
```

**The Prefixes:**
- `,` = Variables and parameters (mutable, runtime values)
- `:` = Type paths (type system namespace)
- `#` = Types, structs, enums (compile-time definitions)
- `|` = Pipelines (executable logic)
- `!` = Errors (exceptional conditions)
- `@` = Registries (organizational metadata)

**The Dot:**
- Universal hierarchy separator
- Same meaning everywhere: "navigate to member"
- No special cases, no context-dependent interpretation
- Works identically whether navigating fields, namespaces, or taxonomies

**Type System Rules:**
- ‚úÖ Use full dot hierarchy: `:pg.array.pg.string` (array of strings)
- ‚úÖ For nested collections: `:pg.array.pg.serial` (array of serialized objects)
- ‚ùå No curly brace syntax: `pg.array{pg.string}` is deprecated
- ‚ùå No nested arrays: No `pg.array.pg.array.T` constructs
- **Rationale:** Maintains universal hierarchy pattern, no special bracket syntax

**Why This Matters:**

1. **Greppability:** Search `,user` finds all variable uses, `|Pipeline` finds all pipeline calls
2. **Tooling:** Syntax highlighters, linters, and IDEs can use simple prefix-based rules
3. **Learnability:** One pattern to learn, applies universally
4. **Composability:** Hierarchies can nest arbitrarily deep without ambiguity
5. **Visual clarity:** Instant recognition of what each identifier represents

**The comma completes the system.** By giving variables their own prefix (`,`), we free the dot (`.`) to serve its single, universal purpose: hierarchy navigation.

This is not just a syntax change‚Äîit's the full realization of Polyglot's core design philosophy.

## Bonus: Universal Alias System

**Problem:** Hierarchies are great for organization but can be verbose.

**Solution:** Explicit alias system using `{A}` blocks!

```polyglot
// Define aliases in standard library
{@} @Polyglot::Types:0.0.0.1
[A] @PgTypes
{x}

{A}
[A] :pg.int >> :int
[A] :pg.string >> :string
[A] :pg.array >> :array
[A] #Boolean.True >> #True
[A] #Boolean.False >> #False
{x}

// Import aliases in your code
{@} @Local::MyApp:1.0.0.0
[A] @App
[<] @PgTypes                           // Import all aliases
{x}

{|} |Pipeline.Example
[<] ,name:string                       // Alias for :pg.string
[<] ,tags:array.string                 // Alias for :pg.array.pg.string

[t] |T.Call
[W] |W.Polyglot.Scope

[r] ,found:bool << #True               // Alias for #Boolean.True
[r] ,items:pg.array.pg.string          // Full form still works!

[>] ,found:bool
{x}
```

**Key features:**
- ‚úÖ **Explicit imports** - `[<] @PgTypes` makes aliases available
- ‚úÖ **Works for all prefixes** - Types, enums, pipelines, errors
- ‚úÖ **Optional** - Don't import if you prefer full names
- ‚úÖ **Composes** - `:array.string` expands to `:pg.array.pg.string`
- ‚úÖ **Both forms work** - Aliased and full names always valid

See `v0.0.4-alias-system.md` for complete documentation.
