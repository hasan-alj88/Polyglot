# Polyglot v0.0.4 - Final Design Decisions

**Date:** 2025-12-11
**Status:** ✅ APPROVED - Implementation Ready

---

## Critical Design Decision: Variable Prefix

### The Problem
Initial proposal used `,` (comma) as variable prefix:
```polyglot
[y] ,age ?[,min_age, ,max_age]         ❌ AMBIGUOUS!
```

**Issue:** Comma is both prefix AND separator - creates visual confusion and parsing ambiguity.

### The Solution
**Use `$` (dollar) as variable prefix:**
```polyglot
[y] $age ?[$min_age, $max_age]         ✅ CLEAR!
```

**Benefits:**
- ✅ No ambiguity with comma separator
- ✅ Familiar from shell, PHP, Perl, PowerShell
- ✅ Clear visual distinction from hierarchy separator (`.`)
- ✅ Single-character, readable, greppable

**Trade-off accepted:** Requires shift key, but this is standard in many languages and the clarity is worth it.

## Complete Prefix System

| Prefix | Purpose | Example | Grep Pattern |
|--------|---------|---------|--------------|
| `$` | Variable/parameter | `$user`, `$user.profile.name` | `grep "\$user"` |
| `:` | Type path | `:pg.string`, `:array.int` | `grep ":pg\."` |
| `#` | Enum/struct | `#OrderStatus.Processing` | `grep "#OrderStatus"` |
| `\|` | Pipeline | `\|Database.Users.Find` | `grep "\|Database"` |
| `!` | Error | `!Network.HTTP.Timeout` | `grep "!Network"` |
| `@` | Registry | `@Local::MyApp:1.0.0.0` | `grep "@Local"` |
| `%` | Metadata | `%Doc`, `%Author` | `grep "%Doc"` |

**Universal pattern:** `PREFIX identifier.hierarchy.path`

## I/O Operator Distinction

### Critical Understanding
Two different contexts for I/O:

**1. Pipeline DEFINITION (signature):**
```polyglot
{|} |MyPipeline
[<] i<input_param:string               // Define input parameter
[>] o>output_param:string              // Define output parameter
{x}
```

**2. Pipeline CALL (usage):**
```polyglot
[r] |MyPipeline
   <input_param << $my_value           // Pass input
   >output_param >> $result_var        // Capture output
```

### Operators
- `i<` - Input parameter (definition)
- `o>` - Output parameter (definition)
- `<` - Input parameter (call)
- `>` - Output parameter (call)

### Markers Always Mean I/O
- `[<]` - Input marker (always about I/O)
- `[>]` - Output marker (always about I/O)

**This distinction prevents confusion between defining a pipeline's signature and calling it.**

## Inline Pipelines (Not "String Literals")

**Terminology correction:** These are "inline pipelines," not "string literals."

### Definition
Special single-output pipelines that evaluate in place:

```polyglot
{|} |Inline.Pipeline.Name
[<] i<formatted_string:string

[t] |T.Call.Inline                     // Special trigger

[W] |W.Implementation

[>] o>output:string                    // MUST be single output
{x}
```

### Usage
```polyglot
[r] $timestamp << |DT.Now""
[r] $doubled << |Double"{$value}"
[r] $safe_sql << |SQL"{$input}"
```

### Inline Error Pipelines
Syntax sugar for error throwing:
```polyglot
!ErrorType"message"                    // Short form

// Expands to:
!ErrorType <message << "message"       // Full form
```

### Planned Inline Pipeline Types
**Backlog for implementation:**
- `|SQL"{...}"` - SQL injection prevention
- `|HTML"{...}"` - HTML entity escaping
- `|JSON"{...}"` - JSON-safe escaping
- `|URL"{...}"` - URL encoding
- `|Regex"{...}"` - Regex escaping
- `|Shell"{...}"` - Shell command escaping
- `|DT.Now""` - Current datetime
- `|Double"{...}"` - Transform operations

## Multi-line Strings: `[+]` Marker

**Design philosophy:** Explicit safeguard against Python's silent concatenation bug.

### Syntax
```polyglot
[r] $query << |SQL""
[+] +"SELECT * FROM orders"
[+] +"WHERE order_id = "
[+] +|SQL"{$order_id}"
[+] +"  AND status = 'pending'"
```

### Why Better Than Triple-Quote
```python
# Python bug (silent concatenation)
items = [
    'item1'     # Forgot comma!
    'item2',
    'item3',
]
# Result: ['item1item2', 'item3']  ❌ SILENT BUG!
```

**Polyglot prevents this:**
- Each line explicitly marked with `[+] +"`
- Can mix literals and inline pipelines
- Parallel evaluation, then safe concatenation
- No silent bugs

## Range Operators (Complete Set)

### Inclusive/Exclusive Variants
```polyglot
[y] $age ?[18, 65]                     // Inclusive: 18 <= age <= 65
[y] $age ?(18, 65]                     // Exclusive min: 18 < age <= 65
[y] $age ?[18, 65)                     // Exclusive max: 18 <= age < 65
[y] $age ?(18, 65)                     // Exclusive both: 18 < age < 65
```

### Collection Membership
```polyglot
[y] $role in? $valid_roles             // In collection
[y] $role !in? $banned_roles           // NOT in collection
```

### Iteration
```polyglot
[p] ~ForEach.Range"1..100"             // Iterate 1 to 100
```

---

## Operator Negation: `!?`

**Universal rule:** All conditional operators with `?` can be negated with `!?`

```polyglot
=?   / !=?                             // Equals / Not equals
>?   / !>?                             // Greater / Not greater (<=)
<?   / !<?                             // Less / Not less (>=)
>=?  / !>=?                            // Greater-equal / Not
<=?  / !<=?                            // Less-equal / Not
?[,] / !?[,]                           // In range / Not in range
in?  / !in?                            // In collection / Not in
re?  / !re?                            // Regex match / Not match
```

**Consistent and predictable across all operators.**

## Variadic Input: `<<<` Shorthand

**Design:** Enable via metadata, not syntax change.

### Pipeline Definition
```polyglot
{|} |Set.Intersection
[<] i<type:pg.type
[<] i<sets:serial                      // Dynamic array
   [%] %InStream << #IO.Stream.TriplePull    // Enable <<<
   [%] %Keys << #IO.Stream.Keys.numeric      // Indexing scheme
   [%] %ItemType << $type                    // Type validation
{x}
```

### Usage
```polyglot
[r] |Set.Intersection
   <<< $set1                           // Appends to array
   <<< $set2
   <<< $set3
   >>> $result                         // Captures output
```

### Metadata Controls Behavior
- `%InStream` - Which shorthand operator (`#IO.Stream.TriplePull` = `<<<`)
- `%Keys` - Indexing scheme (numeric, named, etc.)
- `%ItemType` - Type validation

**Brilliant design:** Metadata makes syntax possible without language-level changes.

## Early Return Pattern

**Using output marker:**
```polyglot
[y] $input =? ""
   [>] o>error << !InvalidInput"Empty input"

[y] $priority !in? $valid_priorities
   [>] o>error << !InvalidOrder"Invalid priority"
```

**Semantics:**
- `[>] o>variable` inside conditional = early return
- Must match a defined output parameter
- Clear and explicit

## Metadata System: `%` Prefix

### Registry Metadata
```polyglot
{@} @Local::MyApp:1.0.0.0
[A] @App

[%] %Doc <<
[+] +"Multi-line documentation"

[%] %Author
   [.] .name << "Alice <alice@example.com>"
   [.] .since << "1.0.0"

[%] %License << "MIT"
{x}
```

### Pipeline Metadata
```polyglot
{|} |MyPipeline
[%] %PublicTo << {@MyApp}
[%] %Doc << "Pipeline documentation"

[%] %Deprecated
   [.] .new_pipeline << |NewVersion
   [.] .message << "Use |NewVersion instead"
   [.] .auto_migrate << #Boolean.True

[<] i<param:string
   [%] %description << "Parameter documentation"
   // %Params.0.name and .type inferred
{x}
```

### Standard Attributes
- `%Doc` - Documentation
- `%Author` - Author (repeatable)
- `%Deprecated` - Deprecation with migration
- `%PublicTo` - Export visibility
- `%Errors` - Expected errors
- `%InStream` - Variadic input config
- `%ItemType` - Type validation

### References
```polyglot
{@MyApp%PrivacyGroup.Auth}             // Metadata reference
@Registry!ErrorType                    // Error from registry
```

## Enum with Value Fields

**More flexible than Rust enums:**
```polyglot
{#} #Result
[.] .success
   [.] .data:serial                    // Value field
   [.] .status:int                     // Value field
[.] .error
   [.] .message:string
   [.] .code:int
{x}
```

**Rule:** Enum field siblings must be enums, but can contain value subfields.

**Hierarchy enables flexible structure.**

## Struct Field Shorthand

**When variable name matches field:**
```polyglot
[r] $name << "Alice"
[r] $email << "alice@example.com"
[r] $timestamp << |DT.Now""

[r] $user << #User
   [.] .name                           // << $name (implicit)
   [.] .email                          // << $email (implicit)
   [.] .created << $timestamp          // Explicit (different name)
```

**Rule:** If no `<<` after field, assumes `<< $<fieldname>`

---

## Struct Auto-filling

**Pipeline can output struct and auto-fill matching fields:**
```polyglot
[r] @OrderData|GetOrder
   <order_id << $order_id
   >record >> $order_record:#OrderRecord    // Auto-fills matching fields

// Then modify specific fields
[r] $order_record << $order_record
   [.] .priority << $new_priority           // Override field
   [.] .updated << |DT.Now""                // Add field
```

---

## Match Expression Exhaustiveness

**Compiler enforces exhaustive matching:**
```polyglot
[m] $result << $value
   [?] "case1" ? "result1"
   [?] "case2" ? "result2"
   [?] * ? "default"                   // Wildcard REQUIRED
```

**Benefits:**
- No unhandled cases
- Compile-time verification
- Safer than if-else chains

---

## File Scope for Enums

**Enums defined at file scope (outside pipelines):**
```polyglot
{@} @Local::Types:1.0.0.0
{x}

{#} #OrderStatus                       // File scope
[.] .pending
[.] .processing
{x}

{|} |UsesEnum                          // Can use enum defined above
{x}
```

**Available throughout the file after definition.**

---

## Collection Literals

**Arrays and sets use `{}` syntax:**
```polyglot
[r] $array:array.int << {1, 2, 3, 4, 5}
[r] $set:set.int << {1, 2, 3, 4, 5}    // Unique values
```

**Type annotation or context determines array vs. set.**

## Summary of All Features

### Syntax Elements
1. ✅ Variable prefix: `$`
2. ✅ I/O distinction: `i<` / `o>` (definition), `<` / `>` (call)
3. ✅ Multi-line strings: `[+]` marker
4. ✅ Inline pipelines: `|Pipeline"{$var}"`
5. ✅ Collection literals: `{1, 2, 3}`
6. ✅ Range operators: `?[,]`, `?(,]`, `?[,)`, `?(,)`
7. ✅ Operator negation: `!?` for all conditionals
8. ✅ Collection membership: `in?`, `!in?`
9. ✅ Early return: `[>] o>variable`
10. ✅ Match exhaustiveness: Compiler-enforced
11. ✅ Enum value fields: Hierarchical structure
12. ✅ Struct shorthand: Field name inference
13. ✅ Struct auto-fill: From pipeline outputs
14. ✅ Metadata system: `%` prefix
15. ✅ Variadic input: `<<<` via metadata
16. ✅ File-scope enums
17. ✅ Block comments: `/* */`
18. ✅ Pipeline composition: `|>`
19. ✅ ForEach.Range: `"1..100"` string literals
20. ✅ Universal hierarchy: `PREFIX.identifier.path`

### Design Philosophy
- **No keywords** - Only markers and operators
- **Explicit over implicit** - Metadata makes intent clear
- **Greppable and toolable** - Unique prefixes for all constructs
- **Safe by default** - Exhaustive matching, explicit multi-line
- **Consistent** - Universal hierarchy pattern everywhere

## Migration from Initial Proposal

**Single breaking change:** `,` → `$` for variables

### Find-Replace Pattern
```regex
# Variable declarations
\[r\] ,(\w+)       →  [r] $\1

# Variable references
,(\w+)             →  $\1

# But preserve:
{,}                →  {,}    (not a variable)
[,]                →  [,]    (not a variable)
```

**All other syntax remains as proposed.**
